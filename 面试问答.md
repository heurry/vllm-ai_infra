# 车企诊断 XML/代码生成平台面试问答

> 更新时间：2026-05-04  
> 当前口径：项目已经从“让 LLM 补 ScriptNode 参数”升级为“检索构建完整证据包，TemplateFamilyResolver 匹配 XML 模板族和流程图说明，LLM 生成每个 ClassName 对应的 TaskNode XML，系统负责校验、汇总、审计和回归”。  
> 注意区分：当前稳定验收闭环仍包含 operation-level `ScriptNode` 生成；新框架已经完成 TemplateFamily 注册、Resolver、Class XML prompt、TaskNode 校验器和 smoke test，目标是逐步替换为 ClassName/TaskNode 级 XML 生成。

---

## 1. 项目一句话介绍

### 30 秒版本

这是一个面向车企诊断开发的 XML/代码自动生成平台。它把 PDF、流程表、历史 XML 模板和流程图说明转成可检索、可回溯的证据包，再通过模板族匹配和 LLM 生成每个诊断节点的 XML，最后用结构校验、证据一致性校验、workflow 汇总、audit 和 regression 保证结果可交付。

### 90 秒版本

车企诊断 XML 的人工开发流程通常是：先从 ECU 参数 PDF 里找到某个 FlowNode 的参数表，再根据 `Refer to the EOL Process-...` 跳到公共流程图，最后参考历史 XML 模板写出当前 ClassName 对应的 TaskNode XML，并在主 workflow 里通过 ScriptNode 引用它。

这个项目把这条人工链路工程化。上游用 MinerU 解析 PDF，得到 Markdown、`content_list.json`、`middle.json` 和图片；Markdown 做语义主干，`content_list` 做页码、bbox、图片路径溯源。然后构建 `KnowledgeUnit` 和 `EvidenceUnit`，对每个 FlowNode 做通用 query 改写、sparse/dense/hybrid 召回、多跳 refer-to 展开、Graph RAG 关系组织和 EvidenceFilter 过滤，形成 `EvidencePack`。

生成阶段不让 LLM 直接写完整 `P177__FHC.xml`，而是让它生成当前 FlowNode 对应 ClassName 的 XML 实现，例如 `ASDM1401_DTC_Clear_Type7.xml`。系统再用 `TaskNodeValidator`、`TemplateFamilyValidator`、`EvidenceConsistencyValidator` 检查 XML 结构、模板族规则和证据来源，最后汇总成总 workflow，并输出 trace、证据链报告、audit 和 regression。

### 3 分钟版本

这个项目解决的问题不是文档问答，而是车企诊断 XML/代码生成。输入来源包括四类：

- `data/processed/mineru/...`：MinerU 从 PDF 解析出的 `.md`、`content_list.json`、`middle.json`、`images/`。
- `flow.xlsx`：定义 8 个流程步骤、11 个具体 FlowNode，以及每个节点引用的 ClassName/TemplateClass。
- `data/xml_template/`：历史人工 XML 模板，例如 `ASDM1401_DTC_Clear_Type7.xml`、`DTC_Clear_Type2.xml`、`GEEA30_VMM_Change.xml`、`P177__FHC.xml`。
- `流程图.md`：人工或 OCR 整理后的流程图文字说明，例如 DTC Clear、DTC Read、IO Control 等流程。

技术链路是：

```text
PDF / flow.xlsx / data/xml_template / 流程图.md
  -> MinerU outputs
  -> Markdown semantic trunk + content_list provenance
  -> KnowledgeUnit / EvidenceUnit
  -> NodeIntent + QueryPlan
  -> sparse + dense + hybrid retrieval
  -> Reference Expansion + EvidenceFilter
  -> Graph RAG / EvidencePack
  -> TemplateFamilyResolver
  -> LLM 生成 ClassName TaskNode XML
  -> TaskNodeValidator / TemplateFamilyValidator / EvidenceConsistencyValidator
  -> 汇总为最终 P177__FHC.xml
  -> trace / audit / regression
```

项目最核心的工程价值是：LLM 不是黑盒写代码，而是在证据、模板和校验器约束下生成；每个参数、每段 XML、每条跳转关系都能回到原始证据。

## 2. 当前项目结构

### 2.1 输入层

```text
data/processed/mineru/...
  MinerU 输出的 md / content_list.json / middle.json / images

pdf-loop/.../flow.xlsx
  当前 workload 的流程步骤和 FlowNode

data/xml_template/
  历史 XML 模板和主 workflow
  例如:
    P177__FHC.xml
    ASDM1401_DTC_Clear_Type7.xml
    ADCU1301_DTC_Clear_Type7.xml
    DTC_Clear_Type2.xml
    DTC_Read_Type1.xml
    GEEA30_VMM_Change.xml

流程图.md
  流程图文字说明
  例如:
    DTC Clear Type7
    DTC Clear Type2
    DTC Read Type1
    IO Control Type21
```

### 2.2 数据处理层

```text
MinerU outputs
  -> Markdown semantic trunk
  -> content_list provenance
  -> SemanticChunk
  -> KnowledgeUnit
  -> EvidenceUnit
```

关键点：

- Markdown 负责语义结构：标题、section、表格内容、流程图说明。
- `content_list.json` 负责版面溯源：`page_idx`、`bbox`、`img_path`、原始 block id。
- `middle.json` 用于底层版面调试，不作为主 RAG chunk。
- `images/` 用于参数页截图、流程图图片和报告展示。

### 2.3 检索和证据层

```text
FlowNode
  -> NodeIntent
  -> QueryPlan
  -> sparse retrieval
  -> Milvus dense retrieval
  -> hybrid merge
  -> reference expansion
  -> evidence filter
  -> Graph RAG
  -> EvidencePack
```

`EvidencePack` 不是简单 topK，而是面向 XML 生成的证据包，包含：

- 当前节点基本信息：node name、ClassName、ECU、operation words。
- 参数证据：参数 section、完整表格、table row、parent table。
- refer-to 路径：从 ECU 参数文档跳到公共流程图。
- 流程图证据：流程标题、流程图文字、流程图图片。
- 模板证据：同族 XML 示例、模板签名、已有 ClassName 写法。
- 过滤记录：keep/demote/drop 原因。

### 2.4 模板族层

离线构建命令：

```bash
.venv/bin/python scripts/build_template_family_registry.py \
  --xml-template-dir data/xml_template \
  --flowchart-markdown 流程图.md \
  --output-path data/processed/template_family_registry/template_families.json \
  --max-example-chars 6000
```

当前 registry 结果：

- `family_count = 6`
- `tasknode_template_count = 7`
- `flowchart_section_count = 5`
- `families_with_flowcharts = 2`

典型模板族：

```text
DTC_Clear_Type7
  aliases:
    ADCU1301_DTC_Clear_Type7
    ASDM1401_DTC_Clear_Type7
    DTC Clear Type7(Single DTC NRC Allowed)

  xml_examples:
    ADCU1301_DTC_Clear_Type7.xml
    ASDM1401_DTC_Clear_Type7.xml

  signature:
    root_tag = TaskNode
    service_id = 0x14
    positive_response = 0x54
    key variables = ClearDTC / AllowedNRC / RetryCount
```

`TemplateFamilyResolver` 在线阶段做：

```text
FlowNode + EvidencePack + template_families.json
  -> ResolvedTemplateFamily
  -> XML examples
  -> flowchart description
  -> expected XML skeleton/signature
```

### 2.5 LLM 生成层

旧稳定路线：

```text
LLM 生成 operation-level ScriptNode / Args
系统汇总成 SerialNode，再融合进 base workflow
```

新框架路线：

```text
LLM 生成当前 ClassName 对应的完整 TaskNode XML
例如:
  ASDM_DTC_Clear
    -> ASDM1401_DTC_Clear_Type7.xml
```

LLM prompt 包含：

- 当前 FlowNode 和 ClassName。
- EvidencePack：完整参数表、refer-to 路径、流程图说明。
- TemplateFamilyResolver 输出的模板族信息。
- 同族 XML 示例。
- XML 输出约束。
- 需要引用的证据 id。

目标输出：

```json
{
  "node_name": "ASDM_DTC_Clear",
  "class_name": "ASDM1401_DTC_Clear_Type7",
  "template_family": "DTC_Clear_Type7",
  "class_xml": "<TaskNode>...</TaskNode>",
  "used_evidence": ["..."],
  "confidence": 0.9,
  "needs_review": false
}
```

### 2.6 校验和汇总层

三层校验：

```text
GenericTaskNodeValidator
  检查 XML well-formed、根节点 TaskNode、MainNodes、CVNodes、FlowsNodes

TemplateFamilyValidator
  按模板族检查结构规律
  例如 DTC_Clear_Type7 检查 0x14、0x54、ClearDTC、AllowedNRC、RetryCount 等模式

EvidenceConsistencyValidator
  检查 XML 中业务值是否来自 EvidencePack
  例如 DTC HEX 是否来自表格 HEX，NRC 是否来自 NRC ALLOWED
```

最终汇总关系：

```text
P177__FHC.xml
  最终总 workflow

ASDM1401_DTC_Clear_Type7.xml
  某个 ClassName 的 TaskNode 实现

ScriptNode
  主流程里引用 ClassName 的入口
```

## 3. MinerU、Markdown 和 content_list 怎么讲

### 3.1 MinerU 不是只生成 Markdown

一个 PDF 解析后通常会得到：

```text
文档目录/
├── xxx.md
├── xxx_content_list.json
├── xxx_middle.json
└── images/
```

各产物分工：

| 产物 | 定位 | 优势 | 局限 | 项目中怎么用 |
| --- | --- | --- | --- | --- |
| `.md` | 语义展示层 | 标题、段落、表格、人类可读 | page/bbox 精确溯源弱 | 做 Markdown semantic trunk |
| `content_list.json` | 版面 block 层 | 有 page、bbox、img_path、block type | text 太细、table 太粗 | 做 provenance anchor |
| `middle.json` | 中间版面结构 | 细粒度 layout 信息 | 复杂，不适合直接检索 | 调试解析问题 |
| `images/` | 原始裁剪图 | 可看表格、流程图、图片区域 | 缺文本语义 | 报告和人工核验 |

面试答法：

> 我不是只处理 Markdown，也不是直接把 MinerU block 当最终 RAG chunk。Markdown 负责语义主干，content_list 负责 page/bbox/img_path 溯源，middle/images 用于调试和原始区域证据。

### 3.2 content_list block 是什么

text block 可能类似：

```json
{
  "type": "text",
  "page_idx": 13,
  "bbox": [267, 749, 791, 762],
  "text": "Refer to the EOL Process-IO Control Type21(Level 3 No Return Control)",
  "text_level": 0
}
```

table block 可能类似：

```json
{
  "type": "table",
  "page_idx": 16,
  "bbox": [120, 210, 1500, 890],
  "table_body": "<table><tr><td>PCODE</td><td>HEX</td>...</tr></table>",
  "img_path": "images/page_016_img_001.jpg"
}
```

为什么不能直接作为最终 chunk：

- text 往往一行一个 block，太细，缺上下文。
- table 往往整张表一个 block，太粗，可能混入很多行。
- 流程图图片需要结合 Markdown 标题或人工/OCR 文本才能有语义。

### 3.3 semantic chunking 怎么做

当前 `build_mineru_index.py --chunking-mode semantic` 会生成：

- `text_chunk`：按 Markdown 标题、段落、页内顺序合并。
- `table_chunk`：从 HTML table 拆出 table parent 和 table row。
- `image_region`：保留流程图/图片区域，并关联标题、caption、附近文本。

每个派生 chunk 保留：

- `source_block_ids`
- `parent_evidence_id`
- `page_idx`
- `bbox` 或 `bbox_union`
- `source_path`
- `doc_id`
- `module`
- `ecu`

最新 semantic index 构建结果：

- `document_count = 10`
- `block_count = 4118`
- `semantic_chunk_count = 7040`
- `knowledge_unit_count = 7040`
- `evidence_unit_count = 7040`
- `image_region = 224`
- `table_chunk = 4580`
- `text_chunk = 2236`

## 4. KnowledgeUnit 和 EvidenceUnit 的区别

### 4.1 不是简单“EvidenceUnit 包含 KnowledgeUnit”

可以这样讲：

- `KnowledgeUnit` 是检索视角，目标是让 sparse/dense/hybrid 更容易召回。
- `EvidenceUnit` 是证据视角，目标是让 XML 参数能回到原始文档、页码、bbox 和 parent table。

二者字段有重叠，但职责不同。

### 4.2 KnowledgeUnit 示例

```json
{
  "chunk_id": "GEEA3.0_EOL_TREG_ADCU1301_Parameter_sem_0553",
  "unit_type": "table_chunk",
  "content": "DTC information as below ... PCODE HEX DESCRIPTION NRC ALLOWED ...",
  "protocol": "UDS",
  "module": "ecu_parameter",
  "ecu": "ADCU1301",
  "service_ids": ["14", "19"],
  "dids": [],
  "page_idx": 16,
  "bbox": [120, 210, 1500, 890],
  "source_path": "..."
}
```

用途：

- 写入 JSONL sparse index。
- 写入 Milvus collection。
- 支持 metadata filter。
- 支持 recall 统计。

### 4.3 EvidenceUnit 示例

```json
{
  "evidence_id": "ADCU1301_sem_0553_tbl_row_4",
  "evidence_type": "table",
  "content": "PCODE=U14D781 HEX=D4D781 DESCRIPTION=... NRC ALLOWED=0x31",
  "doc_id": "GEEA3.0_EOL_TREG_ADCU1301_Parameter-1756876939",
  "source_path": ".../GEEA3.0_EOL_TREG_ADCU1301_Parameter-1756876939.md",
  "page_idx": 16,
  "bbox": [120, 210, 1500, 890],
  "metadata": {
    "block_role": "table_row",
    "parent_evidence_id": "ADCU1301_sem_0553_table",
    "source_block_ids": ["..."],
    "table_fields": {
      "PCODE": "U14D781",
      "HEX": "D4D781",
      "NRC ALLOWED": "0x31"
    }
  }
}
```

用途：

- 给 Graph RAG 构图。
- 给 EvidencePack 和 LLM prompt。
- 给 evidence_chain_report 展示。
- 给证据一致性校验使用。

面试答法：

> KnowledgeUnit 是“怎么被搜到”，EvidenceUnit 是“怎么被证明”。比如一个 DTC 表格行，KnowledgeUnit 保证它能被 `ADCU1301 DTC Clear Type7` 召回，EvidenceUnit 记录它来自哪个 Markdown、哪页、哪个 bbox、parent table 是什么。

## 5. 通用检索策略

### 5.1 为什么不为每个节点写强规则

像 `ADCU1301_DTC_Clear(ADCU1301_DTC_Clear_Type7)`，人工链路是：

```text
找到 ADCU1301 参数文档
  -> 看到 DTC information 表格
  -> 读取 Refer to the EOL Process-DTC Clear Type7
  -> 找到公共流程图 DTC Clear Type7(Single DTC NRC Allowed)
  -> 参考历史 XML 模板生成当前节点 XML
```

但是实际 FlowNode 有几千种，不可能写死：

- 某节点必须命中某个 ECU 文档。
- 某节点必须解析几行表格。
- 某节点必须有固定 request/response。
- 某节点必须两跳就找全信息。

所以当前路线是通用检索、通用跳转、通用过滤和模板族校验。

### 5.2 NodeIntent

对每个 FlowNode 做轻量解析。

输入：

```text
ADCU1301_DTC_Clear(ADCU1301_DTC_Clear_Type7)
```

解析：

```json
{
  "node_name": "ADCU1301_DTC_Clear",
  "template_name": "ADCU1301_DTC_Clear_Type7",
  "target_ecu": "ADCU1301",
  "operation_words": ["DTC", "Clear"],
  "type_tokens": ["Type7"],
  "raw": "ADCU1301_DTC_Clear(ADCU1301_DTC_Clear_Type7)"
}
```

未知节点也不会失败，会 fallback 到 node_name/template_name 原文和通用 token 切分。

### 5.3 QueryPlan

默认生成多条 query：

| Query 类型 | 目的 | 示例 |
| --- | --- | --- |
| anchor query | 锚定原始节点 | `ADCU1301_DTC_Clear ADCU1301_DTC_Clear_Type7` |
| ecu doc query | 找 ECU 参数文档 | `ADCU1301 DTC Clear TREG Parameter` |
| public process query | 找公共流程 | `EOL process DTC Clear Type7` |
| template query | 找历史 XML 模板 | `ADCU1301_DTC_Clear_Type7 XML TaskNode` |
| reference query | 从 refer-to 动态生成 | `EOL Process DTC Clear Type7 Single DTC NRC Allowed` |

这个 QueryPlan 可以继续扩展，例如以后新增：

- routine query
- IO control query
- session/security query
- calibration/programming query
- flowchart title query
- template family query

### 5.4 sparse、dense、hybrid、Graph RAG 的关系

| 模块 | 做什么 | 适合 | 不适合 |
| --- | --- | --- | --- |
| sparse/BM25-like | 关键词召回 | ECU、DID、RID、ServiceID、TypeN、HEX | 同义表达 |
| dense/Milvus | 语义召回 | `dtc read` 和 `read dtc`、同义句、流程描述 | 精确十六进制字段 |
| hybrid merge | 融合 sparse/dense | 去重、score normalize、加权排序 | 解释结构关系 |
| Graph RAG | 组织证据关系 | refer-to、多跳、table parent、flowchart、template | 替代向量检索 |

准确流程：

```text
FlowNode
  -> QueryPlan
  -> sparse/dense/hybrid 初召回
  -> reference expansion
  -> evidence filter
  -> parent context expansion
  -> Graph RAG evidence bundle
  -> LLM prompt
```

面试答法：

> 进入 LLM 的不是 Milvus topK，也不是 sparse topK，而是经过 Graph RAG 和 EvidencePack 编排后的证据包。hybrid 负责找候选，Graph RAG 负责解释和组织候选。

### 5.5 Reference Expansion

通用识别：

```text
Refer to ...
reference to ...
detail refer to ...
can refer to ...
```

每一跳记录：

```json
{
  "hop": 1,
  "from_evidence_id": "...",
  "reference_text": "Refer to the EOL Process-DTC Clear Type7(Single DTC NRC Allowed)",
  "query": "EOL Process DTC Clear Type7 Single DTC NRC Allowed",
  "resolved_title": "DTC Clear Type7(Single DTC NRC Allowed)",
  "resolved_evidence_ids": ["..."],
  "status": "found"
}
```

默认最大跳数 `3`，每跳最多保留 `5` 个候选。

### 5.6 EvidenceFilter

过滤层只做通用相关性和实体一致性，不写节点专用强校验。

| 决策 | 规则 | 说明 |
| --- | --- | --- |
| drop | `explicit_ecu_conflict` | ECU 参数文档明确属于其他 ECU |
| demote/drop | `local_ecu_context_conflict` | section/title/content 出现其他 ECU-like token |
| keep | `public_process_keep` | 公共 EOL process 文档不要求 ECU 匹配 |
| drop | `noise` | 目录、revision log、页眉页脚、空内容 |
| demote | `weak_reference_match` | refer 文本与标题匹配弱 |

所有 keep/demote/drop 都写入 trace。

### 5.7 表格上下文保留

之前发现 DTC Clear 表格只截到第一行，但 `ADCU1301_DTC_Clear_Type7.xml` 需要对表格每一行 HEX 做操作。

当前策略：

```text
命中 table_row
  -> 带 parent table 原文
  -> 带 sibling rows
  -> 带 parent section

命中 section
  -> 保留 section 内完整 table

命中 flowchart title
  -> 带流程图 section 和图片路径
```

`table_fields` 只作为 metadata hint，不作为最终业务答案。

## 6. TemplateFamilyResolver 怎么讲

### 6.1 它解决什么问题

不同 FlowNode 不是都属于 DTC Clear。后续会有：

- DTC Read
- DTC Clear
- IO Control
- Routine Control
- Session Control
- Security Access
- Request Open HV Battery
- Car Mode Change
- Usage Mode Change
- Programming/Calibration 类操作

不能为每一种写一个固定模板生成器。`TemplateFamilyResolver` 的作用是从历史 XML 模板和流程图说明中归纳“模板族”，让 LLM 有可参考的结构，而不是从零写 XML。

### 6.2 离线构建做什么

扫描：

```text
data/xml_template/*.xml
流程图.md
```

生成：

```text
data/processed/template_family_registry/template_families.json
```

每个 `TemplateFamily` 包含：

- `family_id`
- `aliases`
- `xml_examples`
- `flowchart_titles`
- `xml_signature`
- `required_evidence`
- `validation_rules`

### 6.3 在线解析做什么

```text
FlowNode + EvidencePack
  -> TemplateFamilyResolver
  -> ResolvedTemplateFamily
```

它会结合：

- ClassName 文件名，例如 `ASDM1401_DTC_Clear_Type7`。
- FlowNode 名称，例如 `ASDM_DTC_Clear`。
- evidence 中的 refer-to 文本。
- `流程图.md` 中的标题。
- XML 模板签名。

### 6.4 会不会和流程图.md 联动

会。流程图.md 不直接替代 XML 模板，而是作为模板族的流程描述证据：

```text
DTC_Clear_Type7 XML examples
  + 流程图.md: DTC Clear Type7(Single DTC NRC Allowed)
  -> 同一个 TemplateFamily
```

后续补充更多 XML 模板和流程图说明后，Resolver 的覆盖面会扩大。

面试答法：

> TemplateFamilyResolver 不是给每个节点写死模板，而是从 data/xml_template 归纳模板族，再用 flow.xlsx 节点名、refer-to、流程图.md 标题和 XML signature 匹配当前节点应该参考哪一类模板。

## 7. LLM 生成策略

### 7.1 为什么不让 LLM 生成完整 P177__FHC.xml

`P177__FHC.xml` 是最终总 workflow，包含大量已有稳定节点。直接让 LLM 生成完整文件会有问题：

- 上下文太长。
- 局部错误难定位。
- 很难做证据追溯。
- 很难判断哪些是已有稳定逻辑，哪些是新生成逻辑。
- 一处 XML 错误会污染整个 workflow。

所以生成粒度应该是单个 ClassName 的 TaskNode XML。

### 7.2 ScriptNode 和 ClassName XML 的关系

这点面试时要讲清楚：

```text
P177__FHC.xml
  是最终总 workflow

ScriptNode
  是主流程里的调用入口
  里面会引用 ClassName

ClassName XML
  是该节点真正的 TaskNode 实现
  例如 ASDM1401_DTC_Clear_Type7.xml
```

因此新框架的目标不是只生成 ScriptNode，而是生成对应 ClassName 的完整 XML 实现。

### 7.3 LLM prompt 应该包含什么

以 `ASDM_DTC_Clear(ASDM1401_DTC_Clear_Type7)` 为例，prompt 应该包含：

1. 当前节点信息：
   - node name
   - class name
   - template family
   - target ECU

2. 参数证据：
   - ASDM1401 参数文档中的 DTC 表格完整内容。
   - `PCODE / HEX / DESCRIPTION / NRC ALLOWED` 所有行。
   - 不能只给第一行。

3. refer-to 证据：
   - `Refer to the EOL Process-DTC Clear Type7(Single DTC NRC Allowed)`。
   - resolved flowchart title。

4. 流程图说明：
   - 从 `流程图.md` 中匹配到的 DTC Clear Type7 描述。
   - 例如 request/response、retry、NRC 处理逻辑。

5. XML 模板族示例：
   - `ASDM1401_DTC_Clear_Type7.xml`
   - `ADCU1301_DTC_Clear_Type7.xml`
   - 或同 family 的其他 XML 示例。

6. 输出约束：
   - 必须输出完整 `<TaskNode>...</TaskNode>`。
   - 必须保留 XML well-formed。
   - 业务值必须来自 EvidencePack。
   - 输出中要记录 used evidence。

## 8. XML 校验、审计和回归

### 8.1 三类校验

**通用 XML 结构校验**

检查：

- XML well-formed。
- root 是否为 `TaskNode`。
- 是否存在 `MainNodes`。
- 是否存在 `CVNodes`。
- 是否存在 `FlowsNodes`。
- ID/Goto 是否基本可解析。

**模板族校验**

例如 `DTC_Clear_Type7`：

- 是否有 `ClearDTC`。
- 是否有 `AllowedNRC`。
- 是否发送 `0x14`。
- 是否判断 `0x54` 正响应。
- 是否有 RetryCount。
- DTC 表格有多少 HEX，就应该有对应 clear 操作。

注意：这不是为某个 ECU 写死规则，而是模板族级规则。它可以覆盖 ADCU1301、ASDM1401 这类同族节点。

**证据一致性校验**

检查 XML 业务值是否来自 EvidencePack：

- DTC HEX 是否来自参数表 HEX。
- NRC 是否来自 `NRC ALLOWED`。
- DID/RID/ServiceID 是否来自参数表、流程图或模板默认。
- refer-to 标题是否和流程图说明匹配。

### 8.2 Guardrail 的作用

高风险字段：

- `EcuBOMName`
- `DID`
- `RID`
- `ServiceID`
- `RequestParameter`

这些字段必须有来源：

- evidence context
- base workflow
- template contract
- deterministic plan
- explicit semantic default

Trace 记录：

- `raw_llm_args`
- `post_guardrail_args`
- `guardrail_corrections`
- `correction_reason`
- `correction_source`

最近一次回归：

- `raw_llm_critical_arg_accuracy = 0.2`

> raw LLM 准确率低不等于模型没用。LLM 负责理解证据和生成结构，高风险参数由证据、模板和 guardrail 接管。项目真正交付的是 post-guardrail + audit + regression 的结果，而 raw/post 差异会写入 trace。

## 10. 运行态问题怎么回答

### 10.1 vLLM 启动路径问题

如果模型路径包含中文：

```text
/media/xdu/新加卷/LLM_model/Qwen3.6-27B-FP8
```

vLLM/Transformers 可能把本地路径误判成 HuggingFace repo id。解决方式是建 ASCII 软链接：

```bash
ln -sfn "/media/xdu/新加卷/LLM_model/Qwen3.6-27B-FP8" /tmp/Qwen3_5_27B_FP8
```

然后启动：

```bash
.venv/bin/vllm serve /tmp/Qwen3_5_27B_FP8 \
  --served-model-name "/media/xdu/新加卷/LLM_model/Qwen3.6-27B-FP8" \
  --hf-config-path /tmp/Qwen3_5_27B_FP8 \
  --model-impl vllm \
  --host 0.0.0.0 \
  --port 8000 \
  --tensor-parallel-size 2 \
  --max-model-len 32768 \
  --gpu-memory-utilization 0.90 \
  --max-num-seqs 1 \
  --max-num-batched-tokens 8192 \
  --dtype float16 \
  --reasoning-parser qwen3 \
  --language-model-only \
  --trust-remote-code \
  --enforce-eager \
  --enable-prefix-caching
```

### 10.2 Prefix cache hit rate 为 0 正常吗

如果每个节点 prompt 都不同，包含不同 evidence/table/template，那么 prefix cache hit rate 为 0 是正常的。它只说明没有命中相同前缀，不代表生成失败。

判断服务是否正常看：

- `/v1/models` 是否可访问。
- 是否持续有 `POST /v1/chat/completions 200 OK`。
- generation throughput 是否大于 0。
- pipeline 是否写出 prompt、trace、XML。

### 10.3 embedding OOM 怎么处理

问题：

- vLLM 占用 GPU。
- Qwen3-Embedding 又默认上 GPU。
- `build_vector_index` 或 dense query 触发 CUDA OOM。

解决：

- 先停 vLLM，单独用 GPU 构建向量库。
- 或设置 `EMBEDDING_DEVICE=cpu`。
- 或降低 batch size。
- 或把 embedding 服务独立部署。

## 11. 高频面试问答

### 问：这个项目和普通 RAG 有什么区别？

答：普通 RAG 是检索 chunk 后让模型回答；这个项目是证据驱动的 XML/代码生成。检索只是第一步，后面还有 reference expansion、Graph RAG、TemplateFamilyResolver、XML validator、guardrail、workflow assembly、audit 和 regression。最终产物不是自然语言答案，而是可执行 XML 工件。

### 问：BM25 是什么？

答：BM25 是一种关键词检索排序算法，会根据 query 词在文档中的出现频率、稀有程度和文档长度给文档打分。它适合 ECU、DID、RID、ServiceID、TypeN、HEX 这种精确 token，但不擅长同义表达，所以项目里配合 dense retrieval 使用。

### 问：DID 和 RID 是什么？

答：DID 是 UDS 诊断里的 Data Identifier，用于读写某个数据项，例如 `0x3101`。RID 是 Routine Identifier，用于执行例程控制，例如开始某个测试或刷写动作。它们都是诊断 XML 中很关键的参数，但不是所有 FlowNode 都有 DID/RID。

### 问：Milvus 能处理 `dtc read` 和 `read dtc` 吗？

答：可以。Milvus 本身是向量库，语义能力来自 embedding 模型。`dtc read` 和 `read dtc` 经过 embedding 后向量相近，所以 dense retrieval 能补充关键词检索的不足。但十六进制值和 ECU 名称仍然更适合 sparse 精确匹配，所以要 hybrid。

### 问：Graph RAG 和 hybrid retrieval 谁进入 LLM？

答：进入 LLM 的不是原始 hybrid topK，而是 Graph RAG/EvidencePack 编排后的上下文。hybrid 找候选，Graph RAG 建关系，EvidencePack 决定最终给 LLM 的参数表、refer path、流程图和模板示例。

### 问：为什么不用规则直接从表格生成 XML？

答：规则可以做 metadata hint 和校验，但不适合作为全量生成方案。因为节点类型很多，不是都像 DTC Clear 一样有固定表格；有的依赖流程图，有的依赖 Routine/IO Control，有的依赖历史 XML 模板。项目选择通用检索 + 模板族 + LLM 生成 + 校验。

### 问：TemplateFamilyResolver 会不会变成另一套硬编码？

答：不会。它不是为每个 ECU 或每个节点写规则，而是从 `data/xml_template` 自动归纳模板族，从 `流程图.md` 提取流程说明，再按 ClassName、refer-to、XML signature 匹配当前节点。新增模板时重建 registry 即可。

### 问：为什么 raw LLM critical accuracy 只有 0.2？

答：因为 raw LLM 容易在高风险字段上猜值，尤其是 `EcuBOMName`、`DID`、`ServiceID` 等。项目不是直接采用 raw output，而是用 evidence、template contract、deterministic plan 和 guardrail 修正。最终 post-guardrail accuracy 是 1.0，且所有修正都有 trace。

### 问：Guardrail 会不会掩盖模型错误？

答：不会，因为系统同时保存 raw output 和 post-guardrail output。每次修正都有 `correction_source` 和 `correction_reason`，可以看到模型错在哪里、系统为什么修正。

### 问：为什么要 evidence_chain_report？

答：机器 trace 很完整但人不好读。evidence_chain_report 会按每个步骤展示参数来源 Markdown、页码、PDF、参数页图片、Refer 信息、最终流程标题和流程图图片，能快速检查每个节点给 LLM 的证据是否正确。

### 问：如果换车型或换流程怎么泛化？

答：重建 MinerU index、vector index 和 template family registry。核心代码不应该改，因为 NodeIntent、QueryPlan、reference expansion、EvidenceFilter、TemplateFamilyResolver 都是通用逻辑。需要补充的是新车型的 XML 模板和流程图说明。

### 问：当前项目还需要优化什么？

答：三点。第一，把 retrieval golden 从 DID/ServiceID recall 改成 evidence chain coverage。第二，把 ClassName/TaskNode XML 生成端到端跑通并接入 workflow assembler。第三，降低 LLM prompt token 和延迟，避免 embedding 与 vLLM 抢 GPU。

## 12. 实现中遇到的问题

### 12.1 MinerU block 粒度不适合直接 RAG

问题：text 一行一个 block，table 整张表一个 block。

解决：引入 Markdown semantic trunk，把 content_list 作为 provenance，而不是最终 chunk。

面试答法：

> OCR/版面 block 和 RAG chunk 不是一回事。RAG chunk 要服务语义检索和生成，版面 block 要服务溯源。

### 12.2 表格只截取第一行

问题：DTC Clear 需要每行 HEX，但初版 prompt 只带了 table row。

解决：命中 table_row 时回收 parent table、sibling rows 和 parent section。

### 12.3 强校验不可泛化

问题：为 `ADCU1301_DTC_Clear_Type7` 写死表格行数、refer 标题、请求响应，只能跑一个例子。

解决：改成模板族校验。比如 `DTC_Clear_Type7` 是 family 级校验，不绑定 ADCU1301。

### 12.4 检索评测指标设计不合理

问题：golden recall 只关注 DID/ServiceID/RID，但很多节点不需要这些字段。

解决：按节点需要的 evidence 类型评测，例如参数文档、完整表格、refer path、流程图、模板族。

### 12.5 vLLM 与 embedding 抢 GPU

问题：Qwen3.6-27B-FP8 双卡运行时，Qwen3-Embedding 再上 GPU 会 OOM。

解决：离线构建 vector index，或 CPU embedding，或独立 embedding 服务。

### 12.6 Qwen chat template 不支持 developer role

问题：vLLM 返回 `Unexpected message role`。

解决：把 developer prompt 合并进 system prompt，保持语义不变但兼容模型模板。

### 12.7 本地模型路径识别失败

问题：中文路径可能被 HuggingFace 校验当成 repo id。

解决：用 ASCII 软链接启动 vLLM，并设置 `--hf-config-path`、`--model-impl vllm`。

## 13. 简历表达

### 三条项目描述

- 设计并实现车企诊断 XML/代码生成平台，打通 `PDF/MinerU -> semantic evidence -> retrieval/Graph RAG -> TemplateFamilyResolver -> LLM generation -> validation/audit/regression` 闭环，支持从流程表和诊断资料生成可追溯 XML 工件。
- 构建通用证据检索链路：基于 NodeIntent 和 QueryPlan 生成多路 query，结合 sparse/Milvus dense/hybrid 召回、多跳 refer-to 展开、ECU 一致性过滤、parent table 补全和人可读 evidence_chain_report。
- 引入模板族生成路线：从 `data/xml_template` 和 `流程图.md` 构建 TemplateFamily registry，匹配当前 FlowNode 的 XML 模板族，约束 LLM 生成 ClassName 对应 TaskNode XML，并通过结构校验、模板族校验和证据一致性校验保障可交付。

### 技术关键词

```text
MinerU, Markdown semantic chunking, content_list provenance,
KnowledgeUnit, EvidenceUnit, EvidencePack,
BM25-like sparse retrieval, Milvus, Qwen3-Embedding, hybrid retrieval,
Graph RAG, reference expansion, EvidenceFilter,
TemplateFamilyResolver, XML signature, TaskNode XML,
vLLM, Qwen3.6-27B-FP8, prompt trace,
TaskNodeValidator, TemplateFamilyValidator, EvidenceConsistencyValidator,
workflow assembly, audit, regression
```

### 最后总结

> 这个项目最重要的不是“调用大模型写 XML”，而是把诊断资料、检索证据、历史模板、流程图说明、模型生成、XML 校验、workflow 汇总和回归验收连成工程闭环。模型负责理解和生成，系统负责检索、约束、校验、修正和溯源，所以结果不是黑盒生成，而是可解释、可审计、可迭代的 XML/代码生成平台。
