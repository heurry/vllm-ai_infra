# 1. 后续优化总目标

后续不要只说“继续优化 RAG”或者“继续提升生成准确率”，而是要包装成：

> **从单链路诊断 XML 生成系统，升级为可服务化、可观测、可回归、可扩展的 Agentic RAG 代码生成平台。**

优化目标分成 5 个方向：

```text
1. 知识构建更准确：PDF、表格、流程图、XML 模板结构化更稳定
2. 检索证据更可靠：从普通 topK 检索升级为 evidence-type-aware 检索
3. Agent 决策更清晰：Planner / Tool / Memory / Validator 分层
4. XML 生成更可控：从 Prompt 生成升级为模板族约束 + 分阶段生成
5. 工程服务更稳定：API 化、异步任务、资源隔离、监控回归
```

# 3. 第一层：多源数据解析层优化

当前已经有 MinerU 解析、Markdown semantic trunk、content_list provenance、images 溯源。后续优化重点是把“解析”升级成“结构化感知”。

## 3.1 PDF / Markdown 解析优化

当前问题：

```text
Markdown 语义好，但 page/bbox 弱
content_list 有 page/bbox，但 text 太细、table 太粗
流程图图片需要额外结构化
```

后续方案：

```text
PDF
  -> MinerU outputs
  -> Markdown semantic trunk
  -> content_list provenance aligner
  -> table parser
  -> flowchart parser
  -> structured diagnostic knowledge
```

建议新增模块：

```text
src/ingestion/
├── mineru_loader.py
├── markdown_section_parser.py
├── content_list_aligner.py
├── table_normalizer.py
├── flowchart_parser.py
├── xml_template_loader.py
└── flow_xlsx_parser.py
```

## 3.2 表格结构化优化

诊断文档里表格非常关键，比如 DTC、DID、RID、NRC、Request/Response 等。后续不要只保存 table HTML，而是转成结构化字段。

建议输出：

```json
{
  "table_id": "ADCU1301_DTC_Clear_table_001",
  "table_type": "dtc_table",
  "ecu": "ADCU1301",
  "section_title": "DTC Clear Type7",
  "columns": ["PCODE", "HEX", "DESCRIPTION", "NRC ALLOWED"],
  "rows": [
    {
      "PCODE": "U14D781",
      "HEX": "D4D781",
      "DESCRIPTION": "...",
      "NRC_ALLOWED": "0x31",
      "evidence_id": "..."
    }
  ],
  "provenance": {
    "page_idx": 16,
    "bbox": [120, 210, 1500, 890],
    "source_path": "..."
  }
}
```

这样后续 EvidenceConsistencyValidator 可以直接判断：

```text
XML 中的 DTC HEX 是否真的来自表格 HEX
XML 中的 NRC 是否真的来自 NRC ALLOWED
```

## 3.3 流程图结构化优化

当前流程图.md 作为流程说明证据，后续可以进一步转成伪代码/状态机。

目标结构：

```json
{
  "flowchart_id": "DTC_Clear_Type7",
  "title": "DTC Clear Type7(Single DTC NRC Allowed)",
  "nodes": [
    {
      "id": "n1",
      "type": "request",
      "action": "send diagnostic request",
      "service_id": "0x14"
    },
    {
      "id": "n2",
      "type": "condition",
      "condition": "positive response == 0x54"
    },
    {
      "id": "n3",
      "type": "retry",
      "condition": "NRC in allowed list"
    }
  ],
  "edges": [
    ["n1", "n2"],
    ["n2", "n3"]
  ]
}
```

这一步很重要，因为后续 LLM 不只是看自然语言流程图，而是可以根据结构化流程生成 XML 节点。

## 4.3 EvidencePack 优化

后续可以把 EvidencePack 标准化为：

```json
{

    "node_name": "ASDM_DTC_Clear",
    "class_name": "ASDM1401_DTC_Clear_Type7",
    "target_ecu": "ASDM1401",
    "operation": "DTC_Clear"

  "required_evidence_types": [
    "parameter_table",
    "refer_path",
    "flowchart",
    "template_family",
    "xml_example"
  ],
  "parameter_evidence": [],
  "flowchart_evidence": [],
  "template_evidence": [],
  "graph_paths": [],
  "dropped_evidence": [],

    "parameter_table": true,
    "refer_path": true,
    "flowchart": true,
    "template_family": true
  }
}
```

核心优化点是：**EvidencePack 要显式记录“证据是否齐全”**，不能只给 LLM 一堆 context。

# 5. 第三层：检索增强与证据组织优化

当前已经有 NodeIntent、QueryPlan、sparse/dense/hybrid retrieval、reference expansion、EvidenceFilter 和 Graph RAG。后续重点是从“召回 chunk”升级为“按证据类型召回”。

## 5.1 QueryPlan 优化

当前 QueryPlan 包含 anchor query、ECU doc query、public process query、template query、reference query。

后续建议升级成：

```json
{
  "node_name": "ASDM_DTC_Clear",
  "class_name": "ASDM1401_DTC_Clear_Type7",
  "intent": {
    "ecu": "ASDM1401",
    "operation": "DTC_Clear",
    "type": "Type7"
  },
  "queries": [
    {
      "query_type": "parameter_table",
      "query": "ASDM1401 DTC Clear Type7 parameter table",
      "expected_evidence": "table"
    },
    {
      "query_type": "refer_path",
      "query": "Refer to EOL Process DTC Clear Type7",
      "expected_evidence": "reference"
    },
    {
      "query_type": "flowchart",
      "query": "DTC Clear Type7 Single DTC NRC Allowed flowchart",
      "expected_evidence": "flowchart"
    },
    {
      "query_type": "template",
      "query": "ASDM1401_DTC_Clear_Type7 TaskNode XML",
      "expected_evidence": "xml_template"
    }
  ]
}
```

这样检索评估时就可以问：

```text
参数表找到了吗？
refer-to 路径找到了吗？
流程图找到了吗？
模板族找到了吗？
```

而不是只看一个 recall@k。

## 5.2 Hybrid Retrieval 优化

当前 sparse 适合 ECU、DID、RID、ServiceID、TypeN、HEX；dense 适合同义表达和流程描述；Graph RAG 负责组织候选关系。

后续建议采用三阶段检索：

```text
Stage 1: 多路召回
  - BM25 / keyword
  - Milvus dense vector
  - metadata filter
  - template registry lookup

Stage 2: 证据类型重排
  - parameter table reranker
  - flowchart reranker
  - template family reranker
  - reference path reranker

Stage 3: Graph RAG 组织
  - table row -> parent table
  - refer text -> flowchart title
  - class name -> template family
  - template family -> XML examples
```

## 5.3 Reranker 优化

可以新增一个轻量 reranker，不一定马上训练模型，先规则 + LLM judge 都可以。

### 规则 reranker

```text
+ ECU 完全匹配
+ ClassName / TypeN 匹配
+ section title 包含 operation
+ table columns 包含关键字段
+ refer-to 文本命中流程标题
+ XML template signature 匹配
- ECU 冲突
- revision log / header / footer
- 弱相关公共文本
```

### LLM reranker

输入候选 evidence，输出：

```json
{
  "relevance_score": 0.92,
  "evidence_type": "parameter_table",
  "reason": "contains ASDM1401 DTC Clear Type7 table with HEX and NRC columns"
}
```

## 5.4 Context Packing 优化

当前大 prompt 容易导致延迟高。后续要从“尽量多塞”改成“按证据类型压缩”。

建议：

```text
1. 固定系统规则和 XML 约束放 stable prefix
2. 当前节点信息保留完整
3. 参数表只保留必要列和所有关键行
4. 流程图转成结构化步骤，不放长文本
5. XML 示例只保留 signature / skeleton / key snippets
6. dropped evidence 不进 prompt，只进 trace
```

Prompt 结构：

```text
[System Stable Prefix]
你是诊断 XML 生成 Agent...

[Task]
node_name / class_name / template_family / target_ecu

[Evidence Summary]
参数表摘要
refer path
流程图结构
模板族签名

[XML Example Snippets]
只保留关键结构

[Generation Constraint]
必须输出 JSON + TaskNode XML
必须引用 evidence_id
```

## 6.3 Memory 设计

岗位里的 Agent 通常会看“记忆管理”。你的项目可以这样补：

### TemplateMemory

保存：

```text

常见结构
常见变量名
常见 service_id / positive_response
```

### CaseMemory

保存历史成功样例：

```text

```

### FailureMemory

保存失败模式：


### ReviewFeedbackMemory

```text

```

## 6.4 生成策略优化

当前目标是从 operation-level ScriptNode 逐步升级为 ClassName/TaskNode 级 XML 生成。文档中也明确提到，新框架目标是让 LLM 生成当前 ClassName 对应的完整 TaskNode XML，而不是只生成 ScriptNode。

建议采用三阶段生成：

```text
阶段 1：生成 XML Plan
  - 生成节点结构
  - 生成变量清单
  - 生成 evidence mapping

阶段 2：生成 TaskNode Skeleton
  - MainNodes
  - CVNodes
  - FlowsNodes
  - 关键 Goto 关系

阶段 3：填充业务参数
  - ServiceID
  - DTC HEX
  - NRC
  - DID/RID
  - RequestParameter
```

这样比一次性生成完整 XML 更稳。

# 7. 第五层：校验、修复与回归优化

当前已经有 GenericTaskNodeValidator、TemplateFamilyValidator、EvidenceConsistencyValidator。后续应该把校验体系做成分层 Validator Pipeline。

## 7.1 Validator Pipeline

```text
Generated XML
  ↓
SyntaxValidator
  - XML well-formed
  - root tag
  - namespace / encoding

StructureValidator
  - TaskNode
  - MainNodes
  - CVNodes
  - FlowsNodes
  - ID/Goto graph

TemplateFamilyValidator
  - family signature
  - required operation
  - service_id pattern
  - response pattern
  - retry / NRC pattern

EvidenceConsistencyValidator
  - business values grounded in EvidencePack
  - table values match XML
  - flowchart steps match XML logic

WorkflowIntegrationValidator
  - ScriptNode references ClassName
  - ClassName XML exists
  - P177__FHC.xml assembly valid

RegressionValidator
  - compare with baseline
  - detect unexpected diff
  - node golden checks
```

## 7.2 Repair 策略优化

不要所有错误都丢给 LLM 修。建议按错误类型选择修复方式：

| 错误类型                               | 修复方式                        |
| ---------------------------------- | --------------------------- |
| XML 标签未闭合                          | deterministic repair        |
| 缺 root TaskNode                    | deterministic wrapper       |
| 缺 MainNodes / CVNodes / FlowsNodes | LLM repair                  |
| ServiceID 错误                       | evidence/模板覆盖               |
| DTC HEX 缺失                         | 从参数表补全                      |
| ClassName 不匹配                      | TemplateFamilyResolver 重新解析 |
| 流程逻辑缺失                             | 用流程图结构重新生成                  |

## 7.3 Evidence Consistency Score

建议给每个生成结果打一个证据一致性分数：

```text
EvidenceConsistencyScore =
  0.3 * critical_arg_grounding
+ 0.2 * parameter_table_coverage
+ 0.2 * flowchart_alignment
+ 0.2 * template_family_alignment
+ 0.1 * xml_structure_validity
```

结果分层：

```text
>= 0.90 自动通过
0.75 - 0.90 进入人工 review
< 0.75 触发 repair 或重新生成
```

# 9. 性能优化框架

当前瓶颈包括：生成延迟高、prompt token 多、embedding 与 vLLM 抢资源、prefix cache 命中低。文档中已有平均节点生成约 28.6s、总 prompt 约 60937 tokens 的问题描述。

## 9.1 Prompt 压缩

```text
Before:
完整 evidence + 完整 XML 示例 + 完整流程图文本

After:
结构化 Evidence Summary
+ XML signature
+ 必要片段
+ compressed flowchart steps
```

## 9.2 Prefix Cache 优化

把稳定部分放在最前面：

```text
Stable Prefix:
- 系统角色
- XML 输出规范
- 通用校验规则
- 常见错误说明

Dynamic Part:
- 当前 FlowNode
- 当前 EvidencePack
- 当前 TemplateFamily
```

vLLM 启动时可以使用：

```bash
--enable-prefix-caching
```

但前提是 prompt 前缀真的一致。

## 9.3 缓存设计

```text
QueryPlan cache:
  same FlowNode -> same query plan

Retrieval cache:
  same query -> same topK evidence

TemplateFamily cache:
  same ClassName -> same template family

Prompt cache:
  same evidence pack hash -> same prompt

Validation cache:
  same XML hash -> same validation result
```

## 9.4 并发与降级

```text
高优先级任务:
  使用 dense + sparse + graph + LLM

普通任务:
  使用 sparse + template + validator

资源紧张:
  dense retrieval 降级为 BM25
  大模型降级为小模型生成 XML plan
  repair 次数限制
  超时后进入人工 review
```
