# 项目技术方案：车企诊断知识检索与代码生成平台（AI Infra 版）

## 1. 项目概述

### 1.1 项目名称

车企诊断知识检索与代码生成平台

### 1.2 项目定位

本项目面向车企诊断开发场景，建设一个集**异构文档解析、结构化知识入库、混合检索、在线推理服务、结果校验、性能评测与服务治理**于一体的垂直知识推理平台。平台的目标不是单纯完成“代码生成”，而是形成一个**可检索、可验证、可观测、可优化、可部署**的工程化系统。

从能力层次看，项目分为四条主线：

1. **知识工程主线**：将 PDF 协议文档、流程图、历史模板、案例代码等异构资料转为结构化知识单元。
2. **检索增强主线**：构建面向诊断场景的 Hybrid Retrieval 能力，实现可控召回、重排与上下文组织。
3. **推理服务主线**：基于 vLLM 构建在线生成服务，支持单机双卡部署、流式输出、性能优化与服务治理。
4. **结果约束主线**：通过结构化输出、规则校验与回退机制，提高生成结果的可用性与工程可靠性。

### 1.3 项目建设目标

本项目建设目标如下：

- 建立统一的诊断知识建模与结构化入库体系；
- 建设面向车企诊断场景的检索增强生成链路；
- 在单机双卡环境下完成 vLLM 在线推理服务部署与 benchmark 基线测评；
- 建立围绕 TTFT、TPOT、P95、吞吐、cache hit rate 的可观测体系；
- 构建结构化输出与规则校验机制，提升代码生成可验证性；
- 形成一套可持续优化的 AI Infra 工程闭环。

------

## 2. 项目背景与痛点分析

### 2.1 背景

车企诊断开发通常涉及大量协议定义、流程图、会话切换规则、安全访问流程、DID 读写规则以及历史代码模板。这些知识分散在不同载体中，包括：

- PDF 协议文档；
- 图片或扫描版流程图；
- Word / Markdown 技术说明；
- 历史 C 代码模板；
- 人工经验总结与规范约束。

传统方式主要依赖人工查阅文档、复制粘贴模板、手工编写代码，存在开发效率低、知识复用难、输出质量不稳定等问题。

### 2.2 主要痛点

当前场景中的主要问题包括：

#### （1）知识异构且不易复用

文档格式多样，流程图难以直接参与检索与生成，历史代码模板缺乏结构化表达，导致知识无法高效组织与复用。

#### （2）纯文本 RAG 效果有限

协议号、DID、服务号、响应码、流程步骤等知识具有强结构性，仅依赖普通文本切块和向量检索，容易出现召回不准、上下文混乱、生成失真等问题。

#### （3）生成可用性不足

即使模型能输出“看起来合理”的诊断代码，也可能存在以下问题：

- 会话进入顺序错误；
- Security Access 流程不成对；
- 请求 / 响应不符合协议约束；
- 跳转逻辑缺失；
- 输出结构不稳定，不利于自动后处理。

#### （4）缺乏系统性性能评测

在单机双卡部署场景下，若没有统一 benchmark，很难判断：

- 当前瓶颈在检索侧还是推理侧；
- 单卡、双卡 TP、双实例哪种部署更合适；
- 后续优化是否真实有效。

#### （5）缺乏服务治理与可观测能力

缺少延迟、吞吐、显存、缓存命中率、错误率等指标，系统难以持续优化，也难以支撑后续生产级部署。

------

## 3. 总体建设思路

本项目整体遵循如下思路：

> **异构知识结构化 → 混合检索与重排 → 上下文构建 → 在线推理生成 → 结构化校验 → 性能评测与服务优化**

总体上，将系统划分为七大模块：

1. 数据与知识接入模块
2. 文档解析与结构化模块
3. 索引与检索模块
4. Prompt / Context 编排模块
5. 在线推理服务模块
6. 结果校验与回退模块
7. 观测、压测与优化模块

## 5. 关键模块设计

## 5.1 数据与知识接入层

### 5.1.1 输入数据类型

系统第一阶段支持以下输入：

- 协议 PDF 文档；
- 流程图图片 / 截图；
- 历史诊断代码；
- 模板文档；
- Markdown / Word 技术说明；
- 人工整理的规则表。

### 5.1.2 数据接入目标

接入层的目标不是简单保存原始文件，而是为后续结构化处理提供统一入口，做到：

- 数据来源可追踪；
- 文档版本可管理；
- 内容可解析；
- 元数据可过滤。

### 5.1.3 推荐元数据字段

建议为每个文档维护以下 metadata：

```json
{
  "doc_id": "uds_xxx_v1",
  "doc_type": "pdf_protocol",
  "protocol": "UDS",
  "module": "security_access",
  "vehicle_model": "generic",
  "ecu": "BCM",
  "version": "v1.0",
  "source": "internal_upload",
  "created_at": "2026-04-17"
}
```

## 5.2 文档解析与结构化层

这是整个项目最重要的差异化模块之一。目标不是“抽文本”，而是完成**知识建模**。

### 5.2.1 PDF 解析模块

功能包括：

- 文本提取；
- 标题 / 段落识别；
- 表格提取；
- 页码定位；
- 图片区域定位；
- 与流程图检测模块协同。

输出内容：

- 标准文本块；
- 表格块；
- 图片区域；
- 页码与来源映射。

### 5.2.2 流程图结构化模块

针对协议流程图，目标是将图像内容还原为结构化流程步骤。输出可包括：

- 节点类型：开始、处理、判断、结束；
- 请求帧 / 响应帧；
- 判断条件；
- 节点跳转关系；
- 超时 / 失败分支。

示例：

```json
{
  "flow_id": "flow_001",
  "step_id": "step_003",
  "node_type": "process",
  "action": "Enter Extended Session",
  "request": "10 03",
  "response": "50 03",
  "next_step": "step_004",
  "source_page": 12
}
```

### 5.2.3 历史代码解析模块

针对历史诊断代码，抽取其中的结构化模式，包括：

- function_name；
- preconditions；
- step sequence；
- request / response；
- retry policy；
- timeout policy；
- on_fail 逻辑。

目标是把代码模板转化为可供检索和对齐的结构化 DSL。

## 5.3 知识建模与统一 Schema

### 5.3.1 统一知识单元类型

建议系统支持至少三类知识单元：

#### A. 文本块 `text_chunk`

适用于协议说明、描述性内容。

```json
{
  "chunk_id": "txt_001",
  "type": "text_chunk",
  "content": "Security Access Type 1 requires entering extended session first...",
  "module": "security_access",
  "source_page": 8
}
```

#### B. 流程步骤块 `flow_step`

适用于流程图解析结果。

```json
{
  "chunk_id": "flow_001_step_02",
  "type": "flow_step",
  "action": "Request Seed",
  "request": "27 01",
  "response": "67 01 seed",
  "next_step": "flow_001_step_03",
  "module": "security_access"
}
```

#### C. 模板代码块 `code_template`

适用于历史案例或模板函数。

```json
{
  "chunk_id": "tmpl_004",
  "type": "code_template",
  "function_name": "SecurityAccess_Type1",
  "preconditions": ["EnterExtendedSession"],
  "steps": [
    {"send": "10 03", "expect": "50 03"},
    {"send": "27 01", "expect": "67 01 seed"},
    {"send": "27 02 key", "expect": "67 02"}
  ]
}
```

### 5.3.2 Schema 建设目标

统一 schema 的目标包括：

- 统一异构知识表示；
- 支撑不同类型知识的联合检索；
- 支撑结构化校验与回退；
- 为结果渲染和模板生成提供基础。

## 5.4 索引与检索层设计

## 5.4.1 检索架构

检索采用 Hybrid Retrieval 方案，由以下组件构成：

1. **BM25 检索**：召回精确术语、协议号、DID、错误码等；
2. **Dense Retrieval**：召回语义相似案例与说明；
3. **Metadata Filter**：根据车型、协议、ECU、模块等过滤；
4. **Reranker**：对候选结果二次排序；
5. **Context Packer**：将召回结果组织为生成上下文。

### 5.4.2 检索链路

```text
User Query
   ↓
Query Rewrite
   ↓
Hybrid Recall
   ├── BM25
   ├── Dense Retrieval
   └── Metadata Filter
   ↓
Reranker
   ↓
Context Packing
   ↓
Prompt Builder
```

### 5.4.3 Query Rewrite

在诊断场景中，用户问题往往存在自然语言表达不规范的情况，建议增加 Query Rewrite 模块，从请求中抽取：

- 协议类型；
- 功能类别；
- 服务号；
- 目标 ECU；
- 车型限定；
- 预期输出类型。

示例：

原始问题：

> 实现一个 level 5 的信息写入流程

改写后：

> UDS Information Write Type2 level 5, requires extended session and security access level 05, output C-style function template

### 5.4.4 上下文打包策略

上下文组织建议采用优先级机制：

1. 协议 / 规则定义；
2. 结构化流程步骤；
3. 相似历史模板；
4. 描述性解释块。

这样可以减少无关文本对生成的干扰。

## 5.5 Prompt / Context 编排层

### 5.5.1 设计目标

Prompt 层不是手写 prompt 字符串，而是一个可维护的上下文编排模块，目标包括：

- 可配置；
- 可复用；
- 可评测；
- 适合做 prefix cache；
- 适合做 A/B 测试。

### 5.5.2 推荐上下文结构

建议将上下文组织为四段：

#### 段 1：任务描述

明确用户当前需要生成什么。

#### 段 2：规则与协议定义

提供 session、security access、request / response 等规范。

#### 段 3：相似模板与流程步骤

提供最接近的结构化案例和流程。

#### 段 4：输出格式要求

要求输出 JSON / DSL / 伪代码，不直接输出最终 C 代码。

### 5.5.3 输出约束策略

推荐模型先输出中间结构表示：

```json
{
  "function_name": "InformationWrite_Type2_Level5",
  "preconditions": ["EnterExtSession", "SecurityAccess05"],
  "steps": [
    {"send": "10 03", "expect": "50 03"},
    {"send": "27 05", "expect": "67 05 seed"},
    {"send": "27 06 key", "expect": "67 06"},
    {"send": "2E xx ... xx", "expect": "6E xx xx"}
  ],
  "on_fail": "return FAILED"
}
```

再由模板渲染器生成最终代码。

## 5.6 在线推理服务层

这是平台的 AI Infra 核心。

## 5.6.1 服务架构

建议推理服务采用如下分层：

- **API Gateway**：鉴权、限流、请求 ID、超时设置；
- **Router / Request Manager**：请求分类、排队、路由；
- **Generation Worker**：vLLM 实例，负责实际推理；
- **Stream Output Module**：负责流式返回；
- **Cache Layer**：支持检索缓存、prompt / prefix cache；
- **Metrics Exporter**：导出性能指标。

### 5.6.2 单机双卡部署拓扑

当前环境为单机双卡，建议重点评测以下三种形态：

#### 方案 A：单卡单实例

- 作用：作为 baseline；
- 优点：架构简单，便于建立单卡性能基线；
- 缺点：吞吐受限。

#### 方案 B：双卡 Tensor Parallel

- 作用：适合单卡放不下的大模型或长上下文场景；
- 优点：可支持更大模型；
- 缺点：存在跨卡通信开销。

#### 方案 C：双卡双实例

- 作用：每张卡独立运行一个 vLLM 实例，由 Router 分流；
- 优点：通常更适合在线 serving；
- 缺点：需要额外做请求路由与实例治理。

### 5.6.3 推理优化方向

后续优化重点建议包括：

- Retrieval Result Cache；
- Prefix / Prompt Cache；
- Queue 分级；
- 长短请求隔离；
- Chunked Prefill；
- Streaming 输出；
- 多模型路由；
- 参数调优；
- 结果截断与约束。

## 5.7 校验与回退模块

### 5.7.1 校验模块目标

生成不是终点，必须经过校验。校验模块建议分为两层：

#### A. Schema Validator

检查输出 JSON / DSL 是否完整，字段是否齐全。

#### B. Rule Validator

检查诊断流程是否满足规则约束，例如：

- 是否正确进入 Session；
- Security Access 是否成对；
- 请求 / 响应是否匹配；
- 步骤顺序是否合理；
- 是否有完整失败分支。

### 5.7.2 回退机制

当主生成结果不符合要求时，采取兜底策略：

1. 模板回退：使用相似模板填充；
2. Case 回退：直接推荐检索到的历史案例；
3. 规则拼接回退：基于结构化规则拼装最小可用结果；
4. 标记人工复核。

## 6. Benchmark 与性能优化方案

## 6.1 建设原则

在单机双卡场景中，必须先建立一套系统性 benchmark，再逐步优化。原则如下：

1. 先定义 workload，再测 baseline；
2. 端到端、检索侧、推理侧分别测量；
3. 每次优化只改一类关键变量；
4. 用统一 benchmark 框架对比优化前后结果。

## 6.2 Benchmark 测试矩阵

### 6.2.1 输入长度

- 1k tokens
- 4k tokens
- 8k tokens
- 16k tokens

### 6.2.2 输出长度

- 128 tokens
- 256 tokens
- 512 tokens

### 6.2.3 并发数

- 1
- 4
- 8
- 16

### 6.2.4 部署拓扑

- 单卡单实例
- 双卡 TP
- 双卡双实例

## 6.3 基线指标

建议 baseline 包括三层：

### A. 端到端 Baseline

- End-to-End Latency；
- Success Rate；
- P50 / P95 / P99。

### B. 推理侧 Baseline

- TTFT；
- TPOT；
- tokens/s；
- queue wait time；
- GPU 利用率。

### C. 检索侧 Baseline

- retrieval latency；
- rerank latency；
- recall@k；
- prompt length 分布。

## 6.4 优化路线

在建立 baseline 后，按如下优先级推进优化：

### 第一阶段：低成本高收益

- 检索结果缓存；
- Prompt / Prefix Cache；
- Prompt 结构精简；
- Streaming 输出；
- 参数调优。

### 第二阶段：调度优化

- 长短请求分级队列；
- Queue 隔离；
- Chunked Prefill；
- Continuous Batching 参数调优。

### 第三阶段：部署优化

- 双实例替代 TP 验证；
- Router-Worker 架构；
- 多模型路由。

### 第四阶段：高阶优化

- KV cache 复用 / offload；
- Prefill / Decode 解耦实验；
- 多实例动态调度。

## 6.5 Benchmark 结果记录模板

## 7. 实施路径与阶段计划

## 7.1 总体实施节奏

项目建议分八个阶段推进。

### Phase 0：需求与目标冻结

目标：

## 10. 风险分析与应对策略

| 风险点               | 描述                                 | 应对策略                                       |
| -------------------- | ------------------------------------ | ---------------------------------------------- |
| 文档解析质量不稳定   | PDF 结构复杂、流程图质量差           | 建立 schema 化中间结果，支持人工纠错与增量修正 |
| 检索召回不准         | 结构化知识与文本知识混杂，召回不稳定 | Hybrid Retrieval + Metadata Filter + Rerank    |
| 生成结果不可靠       | 规则约束复杂，模型容易 hallucination | 结构化输出 + 规则校验 + Fallback               |
| 单机双卡性能不达预期 | 部署拓扑选择不合理                   | 建立系统 benchmark，对比单卡 / TP / 双实例     |
| 优化不可量化         | 缺少指标与基线                       | 先建立 baseline，再逐项优化                    |
| 服务排障困难         | 缺少链路追踪                         | 引入 tracing 与 metrics 体系                   |

## 13. 下一步建议

最先落地的动作建议按这个顺序推进：

1. 先做 **MVP 全链路**；
2. 立刻建立 **单机双卡 benchmark 基线**；
3. 再补齐 **知识结构化与 Hybrid Retrieval**；
4. 然后做 **缓存、调度、部署优化**；
5. 最后完善 **校验、回退、观测与治理**。

这样做的好处是：
既能尽快跑通系统，又能保证后续每一步优化都有清晰的评测依据和工程价值。
