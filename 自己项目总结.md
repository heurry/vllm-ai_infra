# 车企诊断代码/XML 自动生成平台项目总结与问题回答

## 1. 项目背景

项目来源是吉利 ME 部门的诊断代码自动生成需求。原始流程需要工程师人工翻阅 PDF 文档、流程表和历史 XML workflow：一个 `flow.xlsx` 节点可能对应多个 PDF、多个表格行、多个流程图和多个 `refer to` 目标。人工需要先找到对应参数和流程图，再据此编写或修改诊断代码/XML。

这个流程的问题是：

- 可溯源性差：XML 里只有最终参数值，很难反查来自哪个 PDF、哪页、哪张表、哪一行。
- 参数容易错：比如 ECU、DID、RID、ServiceID、RequestParameter 这类字段相似度高，人工复制容易混淆。
- 难校验：生成结果是否和 base workflow 冲突、是否符合模板契约、是否能通过回归，都缺少自动化闭环。
- 难迭代：换一批车型、换一个日期版本、换一组流程时，人工成本会重复出现。

所以这个项目的目标不是简单“让大模型生成一段 XML”，而是构建一个证据驱动、可校验、可回溯、可回归的诊断代码/XML 生成平台。

整体链路可以概括为：

```text
PDF
  -> MinerU 多产物：Markdown + content_list.json + middle.json + images/
  -> 双层证据构建
     - Markdown semantic evidence：section / table_row / flowchart
     - content_list provenance evidence：page_idx / bbox / block_type / img_path
  -> KnowledgeUnit / EvidenceUnit
  -> sparse + dense + Graph RAG
  -> flow node 级 evidence bundle
  -> LLM node generation + deterministic guardrail
  -> workflow fusion + audit + regression + acceptance
```

说明：当前仓库主链路里，文档解析和规范化主要基于 MinerU 输出；如果面试时提到 Qwen 30B VL，可以说它用于流程图/视觉解析能力探索或辅助解析口径，但最终工程闭环里更重要的是“解析结果如何结构化、索引、检索、校验和追溯”。

MinerU 的输出不是单个 `.md` 文件，而是一套解析产物。更准确的技术口径是：**Markdown 做语义主干，`content_list.json` 做版面溯源和结构补充，`middle.json/images` 做调试和原始区域证据**。

当前主验收链路仍保留 legacy index，作为稳定 baseline；同时已经新增 `--chunking-mode semantic` 的可选索引构建能力：先把 MinerU text/table/image block 处理成 paragraph、table parent、table_row、image/flowchart 等 semantic chunk，再生成 `KnowledgeUnit/EvidenceUnit`，并通过 `source_block_ids`、`page_idx`、`bbox` 回到 `content_list.json` 原始 block。这样可以在不破坏现有验收链路的前提下，对比 legacy chunk 与 semantic chunk 的检索和生成效果。

## 2. metadata 是如何切分的？

这里的“切分”不是传统 RAG 里按 500/1000 token 固定长度切 chunk，而是先区分两类信息：

- **语义信息**：章节、段落、表格行、流程图标题、refer-to 上下文，主要来自 Markdown。
- **版面溯源信息**：页码、bbox、block type、图片路径、原始裁剪区域，主要来自 `content_list.json` 和 images。

所以当前更准确的路线不是“只用 `content_list.json`”，也不是“只用 Markdown”，而是：

```text
Markdown semantic trunk
  -> section / table_row / flowchart evidence
  -> 更适合 Graph RAG 和 LLM prompt

content_list provenance
  -> page_idx / bbox / img_path / block_type
  -> 更适合回到 PDF 原始位置和调试解析问题
```

### 2.1 MinerU 多产物分别是什么？

以当前目录为例，MinerU 对一个 PDF 会输出：

```text
EOL_TREG_EOL_Process-Test_Sequence-1751025478/
├── EOL_TREG_EOL_Process-Test_Sequence-1751025478.md
├── EOL_TREG_EOL_Process-Test_Sequence-1751025478_content_list.json
├── EOL_TREG_EOL_Process-Test_Sequence-1751025478_middle.json
└── images/
```

各自作用如下：

| 产物 | 定位 | 优势 | 局限 | 在项目中的角色 |
| --- | --- | --- | --- | --- |
| Markdown | 语义展示层 | 章节结构清楚，文本连续，适合 section/table_row 切分 | bbox/page 精确溯源弱 | semantic trunk，优先给 Graph RAG 和 LLM evidence |
| `content_list.json` | 版面级 block 层 | 有 `page_idx`、`bbox`、`type`、`img_path` | text block 偏细，table block 偏粗 | provenance anchor，基础 KnowledgeUnit/EvidenceUnit 来源 |
| `middle.json` | 底层版面中间结果 | 有 `pdf_info`、`para_blocks`、`page_size`、discarded blocks | 结构更底层，不适合直接做主索引 | 调试和复杂版面修复 |
| `images/` | 裁剪区域 | 可回看表格/流程图/图片原始区域 | 本身没有语义结构 | 表格、流程图、图片 evidence 的原始证据 |

面试可以这样说：

> MinerU 输出不是单个 Markdown，而是 Markdown、content_list、middle 和 images。Markdown 更适合做语义切分，content_list 更适合做版面溯源。项目的正确路线是 Markdown semantic trunk + content_list provenance，而不是把 content_list block 直接当最终 RAG chunk。

### 2.2 content_list block 具体是什么格式？

MinerU 解析 PDF 后，会把页面内容拆成一组 block。项目里把它简化成 `MinerUContentBlock`，核心字段是：

```json
{
  "type": "text | table | image | equation",
  "page_idx": 4,
  "bbox": [267, 749, 791, 762],
  "text": "2.3.1.3.10.23 Routine Control Type18(Level5 Without 3101)119",
  "text_level": 0,
  "table_caption": [],
  "table_body": "",
  "table_footnote": [],
  "image_caption": [],
  "image_footnote": [],
  "img_path": ""
}
```

不同类型的 block 取内容的方式不同：

| block 类型 | content 怎么来 | 典型 metadata |
| --- | --- | --- |
| `text` | 使用 `text` 字段 | `page_idx`、`bbox`、`text_level` |
| `table` | `table_caption + table_body + table_footnote` 拼接 | `page_idx`、`bbox`、表格图片路径 |
| `image` | `image_caption + image_footnote` 拼接，必要时保留 `img_path` | `page_idx`、`bbox`、`img_path` |
| `equation` | 使用公式文本 | `page_idx`、`bbox` |

举一个真实索引里的例子，一个 MinerU text block 最后会形成类似内容：

```json
{
  "content": "2.3.1.3.10.23 Routine Control Type18(Level5 Without 3101)119",
  "page_idx": 4,
  "bbox": [267, 749, 791, 762],
  "source_path": "data/processed/mineru/.../_content_list.json"
}
```

这里 `page_idx` 表示 PDF 页序号，`bbox` 是该 block 在页面上的坐标框，`source_path` 指向 MinerU 的原始解析文件。这三个字段不是业务参数，但对后续追溯很重要。

但要注意：`content_list.json` 是版面级解析结果，不是最终最佳 RAG chunk。

- text block 通常偏细，可能一行就是一个 block，语义上下文不足。
- table block 通常偏粗，可能整张表是一个 block，容易混入很多无关参数。
- 所以它适合做 provenance anchor，不适合独立承担最终 evidence chunk。

### 2.3 Markdown semantic trunk 怎么切？

Markdown 的优势是已经把很多行自然组织成可读结构。当前项目用 Markdown 构建更适合 Graph RAG 和 LLM 的 EvidenceUnit：

- section evidence：按 Markdown 标题和内容切成章节级证据。
- table_row evidence：解析 HTML/Markdown 表格，把整表拆到行级，并保留 `table_fields`。
- flowchart evidence：识别流程图标题、图片引用和 `refer to` 文本。

例如 table row evidence 会保留：

```json
{
  "evidence_id": "..._tbl_1_1_2",
  "evidence_type": "table",
  "content": "section title\nDID: 0x3101 REQUEST PARAMETER: 0x00",
  "metadata": {
    "source": "mineru_markdown",
    "block_role": "table_row",
    "section_title": "...",
    "parent_evidence_id": "..._sec_0001",
    "table_fields": {
      "DID": "0x3101",
      "REQUEST PARAMETER": "0x00"
    }
  }
}
```

这种粒度比 `content_list` 的一行 text 或整张 table 更适合进入 Graph RAG 和 LLM prompt。

### 2.4 从 block 到 metadata 的切分规则

可以把 metadata 分成三层：

| 层级 | 字段 | 来源 | 作用 |
| --- | --- | --- | --- |
| 文档级 metadata | `doc_id`、`module`、`ecu`、`protocol`、`source_path` | 文件名、目录名、workload 默认配置 | 判断这条内容属于哪个文档、哪个 ECU、哪个模块 |
| 位置级 metadata | `page_idx`、`bbox`、`img_path` | MinerU block | 回到原文页面和区域 |
| 内容派生 metadata | `service_ids`、`dids`、`sessions`、`security_levels` | 从 content 中用规则抽取 | sparse 检索、过滤、参数候选 |

例如文件名里有：

```text
GEEA3.0_EOL_TREG_ADCU1301_Parameter-1756876939
```

系统会推断：

```json
{
  "doc_id": "GEEA3.0_EOL_TREG_ADCU1301_Parameter-1756876939",
  "module": "ecu_parameter",
  "ecu": "ADCU1301",
  "protocol": "UDS"
}
```

如果 block 内容里出现：

```text
Routine Control Type18(Level5 Without 3101)
```

规则会抽取出：

```json
{
  "service_ids": ["10"],
  "dids": ["3101"],
  "security_levels": ["05"]
}
```

这里要注意：这种规则抽取是为了检索加权和候选过滤，不等于最终 XML 参数一定采用这些值。最终高风险参数还要过 evidence、template contract、guardrail 和 audit。

### 2.5 KnowledgeUnit 怎么切

`KnowledgeUnit` 是检索用的知识块。当前实现里，它主要保留了 `content_list.json` 的原始 block 粒度，所以适合作为 baseline 初召回和稀疏/向量检索的基础输入。

当前实现的切分逻辑大致是：

- MinerU 输出里每个 text/table/image/equation block 会变成一个候选知识块。
- text/equation 使用文本内容。
- table 使用表格 caption、table body、footnote 拼接后的内容。
- image 使用图片 caption、footnote。
- 每个块保留 `page_idx`、`bbox`、`source_path`。
- 从文档路径或文件名推断 `doc_id`、`module`、`ecu`。其中 `KnowledgeUnit` schema 里没有单独的 `doc_id` 字段，`doc_id` 会体现在 `chunk_id` 前缀和 `source_path` 中。
- 从内容中用规则抽取 `service_ids`、`dids`、`sessions`、`security_levels`。

但更优路线里，`KnowledgeUnit` 不应该只来自原始 block，而应该优先来自 semantic chunk：

- paragraph chunk：由连续 text block 合并得到。
- section summary chunk：由 Markdown section 得到。
- table row text chunk：由表格行字段拼接得到。
- flowchart title/context chunk：由流程图标题和引用关系得到。

同时保留 `source_block_ids`、`page_idx`、`bbox_union`，把语义 chunk 关联回 `content_list.json` 原始 block。

真实 `KnowledgeUnit` 示例：

```json
{
  "chunk_id": "EOL_TREG_EOL_Process-Test_Sequence-1751025478_0049",
  "unit_type": "text_chunk",
  "content": "2.3.1.3.10.23 Routine Control Type18(Level5 Without 3101)119",
  "protocol": "UDS",
  "module": "eol_process",
  "ecu": null,
  "service_ids": ["10"],
  "dids": ["3101"],
  "sessions": [],
  "security_levels": ["05"],
  "page_idx": 4,
  "bbox": [267, 749, 791, 762],
  "source_path": "data/processed/mineru/.../_content_list.json"
}
```

所以 `KnowledgeUnit` 更适合做检索，因为它把正文和可检索 metadata 放在一起。

面试可以这样说：

> 当前实现里 KnowledgeUnit 保留了 content_list 的原始 block 粒度，适合做 baseline 初召回；但更优的路线是先做 semantic chunking，把 text line 合并成 paragraph，把 table 拆成 row，再生成 KnowledgeUnit。这样检索粒度更合理，同时通过 source_block_ids、page_idx、bbox 仍然能回到 MinerU 原始 block。

### 2.6 EvidenceUnit 怎么切

`EvidenceUnit` 是追溯用的证据块，它和 `KnowledgeUnit` 的区别是更强调“这个参数来自哪里”。

EvidenceUnit 的来源有两类：

- content_list provenance evidence：每个原始 block 转成基础证据，保留 `evidence_id`、`doc_id`、`page_idx`、`bbox`、`source_path`、`img_path`。
- Markdown semantic evidence：把 Markdown 的 section、table row、flowchart title、flowchart image 等拆成更适合 Graph RAG 和 LLM 的证据。

对于表格，项目会尽量拆到 table row 级别，并保留表头字段，例如 DID、RID、Request Parameter、Source、Target 等。这样最终某个 XML 参数可以追溯到具体表格行，而不是只追溯到一整个 PDF。

更优路线里，EvidenceUnit 应该同时保存两类关系：

- 语义关系：`parent_evidence_id`、`section_title`、`table_fields`、`flowchart_title`。
- 版面关系：`source_block_ids`、`page_idx`、`bbox_union`、`img_path`。

真实 `EvidenceUnit` 示例：

```json
{
  "evidence_id": "EOL_TREG_EOL_Process-Test_Sequence-1751025478_ev_0019",
  "evidence_type": "text",
  "content": "2.3.1.3.9.10 IO control Type14(Level11 No return control No read DID)93",
  "doc_id": "EOL_TREG_EOL_Process-Test_Sequence-1751025478",
  "source_path": "data/processed/mineru/.../_content_list.json",
  "page_idx": 4,
  "bbox": [218, 301, 841, 315],
  "metadata": {
    "doc_type": "pdf_protocol",
    "protocol": "UDS",
    "module": "eol_process",
    "ecu": "",
    "source": "mineru",
    "block_role": "section_text",
    "service_ids": ["10"],
    "dids": [],
    "sessions": [],
    "security_levels": ["11"]
  }
}
```

Markdown 表格行会更细，例如：

```json
{
  "evidence_id": "..._tbl_1_1_2",
  "evidence_type": "table",
  "content": "document ... Document No 00000000 ... Revision",
  "metadata": {
    "source": "mineru_markdown",
    "block_role": "table_row",
    "section_title": "document",
    "parent_evidence_id": "..._sec_0001",
    "table_fields": {
      "GEELY GEELY AUTOMOTIVE RESEARCH INSTITUTE CO.LTD": "Document No 00000000",
      "DOCUMENT RELEASE STATUS RELEASED": "Revision"
    }
  }
}
```

面试可以这样说：

> EvidenceUnit 不只是把 content_list block 复制一遍，而是要把 Markdown 的语义证据和 content_list 的版面锚点结合起来。LLM 看到的是 section/table_row/flowchart 这些语义证据，audit 和 trace 还能通过 page_idx、bbox、source_block_ids 回到原始 PDF 区域。

### 2.7 为什么不只用 Markdown，也不只用 content_list？

只用 Markdown 的问题：

- 语义结构好，但 page/bbox/img_path 溯源弱。
- 对表格和流程图的原始区域定位不够直接。
- 解析错误排查时，很难回到 MinerU 的底层 block。

只用 `content_list.json` 的问题：

- text block 太细，dense embedding 和 LLM 上下文都容易缺语义。
- table block 太粗，整表召回会混入很多无关行。
- flowchart 和 refer-to 上下文不如 Markdown section 清楚。

推荐路线：

```text
Markdown 负责语义主干：section / table_row / flowchart
content_list 负责版面溯源：page_idx / bbox / img_path / block_type
middle.json/images 负责底层调试和原始区域证据
```

面试可以这样回答：

> 我不会只处理 Markdown，也不会直接把 content_list 当最终 chunk。Markdown 更适合做 semantic trunk，content_list 更适合做 provenance anchor。真正好的 RAG 输入应该是语义 chunk，同时保留 source_block_ids、page_idx、bbox，这样既让模型读得懂，又能让结果追得回。

### 2.8 已落地的第一阶段：SemanticChunkPreprocessor

当前已经在 `content_list.json` 和 `KnowledgeUnit/EvidenceUnit` 之间增加了第一版 `SemanticChunkPreprocessor`，通过 `scripts/build_mineru_index.py --chunking-mode semantic` 启用。它不是替换原 baseline，而是作为可选实验路径接入 workload。

已实现的处理逻辑：

- text block 合并：
  - 按同页、block 类型和连续顺序合并 MinerU 过细的文本行。
  - 生成 paragraph/section chunk。
  - 保留 `source_block_ids` 和 `bbox_union`。
- table block 拆行：
  - 解析 HTML table 的 `tr/td/th`。
  - 生成 table parent chunk 和 table_row chunk。
  - 每个 table_row 保存 `table_fields`、`row_index`、`parent_evidence_id`。
- image/flowchart block 对齐：
  - 保留 image evidence。
  - 关联 Markdown flowchart title 和 `refer to` 文本。
  - 保留 `img_path`、`page_idx`、`bbox`。

落地后的可选链路：

```text
content_list raw blocks + Markdown sections
  -> SemanticChunkPreprocessor
  -> semantic chunks with provenance
  -> KnowledgeUnit / EvidenceUnit
  -> sparse + dense + Graph RAG
```

对应入口：

```bash
python scripts/build_mineru_index.py \
  --mineru-output-dir data/processed/mineru/treg_20260402_api/extracted \
  --index-dir data/index/treg_20260402_semantic \
  --collection-name treg_20260402_semantic \
  --chunking-mode semantic
```

也可以通过 `configs/workloads/treg_20260402_semantic.json` 让 pipeline 先构建 semantic index，再继续 deterministic XML plan、fusion、audit 和 regression。

当前这条 semantic workload 已经跑通：

- semantic index：10 个文档、4118 个 MinerU block、7040 个 semantic chunk。
- KG：17071 个 graph entity、30744 条 graph relation、18938 个 KG node、39768 条 KG edge。
- regression：`valid=true`，8 个 flow step、11 个 planned node、11 个 operation XML 全部合法。
- trace 覆盖：operation retrieval coverage = 1.0，operation graph path coverage = 1.0。
- node golden：11/11 通过，accuracy = 1.0。
- audit：review items = 4，主要是 base workflow 与 plan 冲突保留 base 的审计项。

继续接入 Milvus 后，`configs/workloads/treg_20260402_semantic_hybrid.json` 也已经跑通：

- pipeline：`valid=true`。
- Milvus collection：`diagnostic_knowledge_treg_20260402_semantic_hybrid`。
- vector count：14080，其中 7040 条 KnowledgeUnit、7040 条 EvidenceUnit。
- embedding dimension：1024，metric：COSINE。
- operation dense retrieval coverage：1.0。
- operation hybrid retrieval coverage：1.0。
- operation XML valid：11/11。
- node golden：11/11，accuracy = 1.0。
- audit review items：4。
- average dense latency：约 0.84s。
- retrieval golden dense/hybrid recall@8：0.333333。

这里要特别说明：semantic chunk 改变了证据 ID 体系，旧的 retrieval golden set 仍然按 legacy evidence id 评测。现在已经把 `source_evidence_id` 加入 Milvus schema、dense trace 和 regression ranking，避免 trace 丢失；但剩下的 0.333333 说明 semantic chunk 的排序与旧 golden 仍不完全一致，下一步需要做 semantic 专用 golden set、chunk 粒度调优或 reranker，而不是简单认为链路不可用。

后续还可以继续优化：

- 用 Markdown 标题层级辅助 text block 合并，减少跨章节误合并。
- 用 Markdown 表格语义补齐 HTML table row 的列名和上下文。
- 把 Markdown flowchart title 与 content_list image block 做更稳定的邻近对齐。
- 对 dense 检索做更细的过滤和重排，避免无 ECU 过滤时被相似但错误的 ECU 参数表抢占 topK。

### 2.9 为什么已经有 KnowledgeUnit，还要有 EvidenceUnit？

你看到的疑问是对的：`KnowledgeUnit` 里也有 `page_idx`、`bbox`、`source_path`、`service_ids`、`dids`，看起来也能追溯。但两者职责不一样。

可以用一句话区分：

```text
KnowledgeUnit 是为了“搜得快、排得准”；
EvidenceUnit 是为了“说得清、审得过、追得回”。
```

具体区别：

| 对比项 | KnowledgeUnit | EvidenceUnit |
| --- | --- | --- |
| 主要用途 | 检索和过滤 | 证据追溯、LLM prompt、audit trace |
| ID | `chunk_id` | `evidence_id` |
| doc_id | 不单独存字段，通常在 `chunk_id` 前缀里 | 单独有 `doc_id` 字段 |
| metadata 形态 | 扁平字段，方便 BM25、filter、embedding 拼接 | 嵌套 metadata，方便保存 block_role、table_fields、parent_evidence_id、flowchart_title |
| 粒度 | 更接近 MinerU block | 可以是 MinerU block，也可以是 Markdown section/table row/flowchart |
| 面向对象 | retrieval engine | generation、guardrail、audit、trace |

为什么两个对象都要保留 `page_idx`、`bbox`、`source_path`？

- 因为检索阶段也需要这些字段返回给用户或传给后续模块。
- 但在 `KnowledgeUnit` 里，它们主要是检索结果 metadata。
- 在 `EvidenceUnit` 里，它们是证据链的一部分，会进入 `evidence_map`、`prompt_context_ids`、`guardrail_corrections` 和 audit trace。

面试可以这样回答：

> 两者字段有重叠，是故意设计的，不是重复建模。KnowledgeUnit 面向检索，所以把 DID、ServiceID、ECU 等字段扁平化，方便 sparse/dense scoring 和 filter；EvidenceUnit 面向审计和追溯，所以保留 evidence_id、doc_id、block_role、table_fields、parent evidence、flowchart 等更丰富的 provenance。检索可以命中 KnowledgeUnit，但最终生成和审计必须落回 EvidenceUnit。

更贴近当前优化思路的回答是：

> 更准确的路线不是只处理 Markdown，也不是直接把 content_list 当最终 chunk，而是 Markdown 做语义主干，content_list 做版面溯源。KnowledgeUnit/EvidenceUnit 最好基于语义 chunk 构建，同时通过 source_block_ids、page_idx、bbox 回到 MinerU 原始 block。

## 3. JSONL sparse index 的识别原理是什么？

JSONL sparse index 本身不是一个复杂数据库，它的本质是本地文件索引：

- `knowledge_units.jsonl` 保存所有 `KnowledgeUnit`。
- `evidence_units.jsonl` 保存所有 `EvidenceUnit`。
- `summary.json` 保存统计信息。

sparse 检索时会读取 `knowledge_units.jsonl`，然后做一个轻量 BM25 风格的打分。

核心原理：

1. 对 query 和文档内容做 tokenization。
2. token 规则会特别识别：
   - `0x3101` 这类十六进制值。
   - `ADCU1301` 这类 ECU 名。
   - `ServiceID 0x19` 这类服务号。
   - 普通英文/数字 token。
3. 计算 BM25-like 分数：词频越高、词越稀有，分数越高。
4. 额外加 exact match boost：
   - query 完整出现在 content 中会加分。
   - query token 命中 unit 的 `dids`、`service_ids`、`sessions`、`security_levels` 会加分。
5. 支持 metadata filter，例如 ECU、DID、module、unit_type。

所以它对 `0x3101`、`ADCU1301`、`ServiceID 0x19` 这种精确标识符很有效。

面试可以这样说：

> JSONL sparse index 的原理是本地 BM25 + 业务字段 boost。它不是语义检索，而是对诊断场景里的强标识符做精确匹配，比如 ECU、DID、ServiceID。dense 检索可能对自然语言更强，但这类十六进制参数和 ECU 编号，sparse 反而更稳定。
BM25 是一种 sparse retrieval 算法，本质是关键词相关性排序。它综合考虑 query 词在文档中的出现次数、这个词在全库里的稀有程度，以及文档长度归一化。在诊断代码生成项目里，BM25 很适合召回 ECU、DID、ServiceID 这类强标识符，所以我把它作为 sparse baseline，再用 Milvus dense retrieval 补充语义召回。
BM25 = 词频 + 稀有度 + 文档长度归一化
query
  -> 分词
  -> 和 knowledge_units.jsonl 里的每个 KnowledgeUnit 做关键词匹配
  -> 计算 BM25 分数
  -> 再对 DID / ServiceID / session / security_level 做业务加权
  -> 返回 topK

关键词检索不只有 BM25。最基础的是 Boolean 和 TF-IDF，工业界常用 BM25/BM25F，工程系统里通常是 Lucene/Elasticsearch 的 BM25 加字段权重和 filter。更先进的是 SPLADE、DeepImpact 这类 neural sparse retrieval。当前项目数据规模不大，而且诊断文档里有大量 ECU、DID、ServiceID 这类强标识符，所以我先用 BM25 + exact boost + metadata filter 做稳定 baseline，后续如果规模上来可以迁移到 BM25F 或 Elasticsearch。
BM25F	BM25 的多字段版本	可区分 title、body、metadata 权重
你的项目当前更接近：

BM25 + exact match boost + metadata filter
也就是：

BM25 负责基础关键词相关性。
exact boost 强化 DID、ServiceID、session、security_level。
metadata filter 支持按 ECU、module、unit_type 等过滤。
## 4. Milvus 向量库的作用是什么？

Milvus 的作用是承载 dense retrieval，也就是语义向量检索。

sparse 检索擅长精确匹配，但如果 query 和文档表达方式不完全一样，就可能漏召回。例如：

- flow 里写 `Request Open HV Battery`。
- 文档里可能写 `Request BECM to Open HV Battery`。
- sparse 不一定每个词都命中，但 embedding 能捕捉语义相似性。

因此 Milvus 不是替代 sparse，而是补充 sparse。

当前真实构建结果：

- collection：`diagnostic_knowledge_treg_20260402`
- embedding model：`/home/xdu/LLM/models/Qwen3-Embedding-0.6B`
- vector count：`11140`
- embedding dimension：`1024`
- metric：`COSINE`
- knowledge units：`4019`
- evidence units：`7121`

面试可以这样说：

> Milvus 在这里是语义召回通道，用来补 sparse 漏掉的语义相关证据。但它不是事实源，事实源仍然是 JSONL 和原始文档。Milvus 返回结果后必须带回 source id、page、bbox、source path，才能进入 evidence trace。
Milvus 本身负责向量近邻检索，真正让 DTC Read 和 Read DTC 靠近的是 embedding 模型。它能处理词序变化和同义表达，例如 DTC Read、Read DTC、read diagnostic trouble code。但对 DID、ECU、ServiceID 这种精确编号，sparse BM25 更稳定，所以项目采用 hybrid retrieval：dense 补语义召回，sparse 保证强标识符命中。

## 5. 构建向量库的数据来源是什么？

向量库的数据来源是已经构建好的本地索引目录：

- `data/index/treg_20260402/knowledge_units.jsonl`
- `data/index/treg_20260402/evidence_units.jsonl`
- `data/index/treg_20260402/summary.json`

构建向量库时，系统会读取 `KnowledgeUnit` 和 `EvidenceUnit`，把每条记录转换成 `VectorRecord`：

- `primary_id`：区分知识单元或证据单元，例如 `ku:<chunk_id>`、`ev:<evidence_id>`。
- `source_id`：原始 `chunk_id` 或 `evidence_id`。
- `record_type`：`knowledge_unit` 或 `evidence_unit`。
- `content`：正文 + 关键 metadata 拼接后的文本。
- `metadata`：unit_type、evidence_type、doc_id、module、ecu、service_ids、dids、source_path、page_idx、bbox 等。

然后用 embedding 模型把 content 编码成向量，再写入 Milvus。

面试可以这样说：

> 向量库不是重新解析 PDF，而是从已经规范化的 JSONL 索引构建。这样可以保证 dense 召回结果能映射回原始 KnowledgeUnit 或 EvidenceUnit，也能通过 manifest 校验源文件 hash、写入数量、向量维度和 collection schema。

## 6. 为什么要使用 embedding 模型？

embedding 模型的作用是把文本映射到向量空间，让语义相近的文本距离更近。

在这个项目里，query 可能来自 flow node，例如：

```text
ECM1610_Request_Open_HV_Battery
```

文档里的证据可能是：

```text
Request BECM to Open HV Battery
```

如果只做关键词匹配，两者不一定完全命中；embedding 可以把它们映射为相近向量，再通过 Milvus 找最近邻。

但 embedding 不能解决所有问题：

- 对 `0x3101` 这类精确 DID，sparse 更稳。
- 对 flowchart refer-to 路径，Graph RAG 更稳。
- 对最终参数是否可接受，guardrail 和 audit 更稳。

所以 embedding 是召回增强，不是最终决策。

面试可以这样说：

> embedding 解决的是语义相似召回，不负责最终判断。最终是否接受还要看 evidence id、template contract、deterministic guardrail 和 workflow audit。

## 7. hybrid retrieval merge 里的 dense 和 sparse 是在哪里做的？原理是什么？

dense 和 sparse 的入口在本地检索接口里：

- sparse：读取 `knowledge_units.jsonl`，做 BM25-like 检索。
- dense：调用 `query_dense_index`，把 query 通过 embedding 模型转成向量，再查 Milvus。
- hybrid：在 `query_local_index` 中合并 sparse 和 dense 结果。

hybrid merge 的原理：

1. 分别得到 sparse chunks 和 dense chunks。
2. 对两路 score 分别做 min-max normalization。
3. 按 chunk id 去重。
4. 加权合并：
   - dense weight：当前 workload 是 `0.6`
   - sparse weight：当前 workload 是 `0.4`
5. 如果同一个 chunk 同时被 dense 和 sparse 命中，source 会标记为 hybrid，并保留 `hybrid_sources`。
6. 最终取 topK。

进入 Graph RAG evidence bundle 时，还会把检索结果重新映射回 EvidenceUnit：

- 如果 chunk metadata 里有 `source_id`，优先用 source_id 找原始 EvidenceUnit。
- 找不到时，会用 retrieved chunk 临时构造 EvidenceUnit，但仍保留 source metadata。

面试可以这样说：

> hybrid retrieval 不是简单拼接两个列表，而是先分别打分、归一化、按 id 去重，再按权重融合。这样既能利用 sparse 的精确匹配，也能利用 dense 的语义召回，并且每条结果仍能映射回原始 evidence。

## 8. Graph RAG 是什么时候构建的？是为每个 flow node 单独构建一个吗？

Graph RAG 的图谱不是每个 flow node 单独构建一次，而是在索引构建阶段基于全量 EvidenceUnit 构建一个全量知识图谱/诊断图谱快照。

也就是说：

- 构建索引时：从 MinerU/Markdown/EvidenceUnit 中抽取文档、章节、表格行、参数字段、ECU、DID、RID、ServiceID、流程图等实体和关系，形成全局 graph。
- 查询 flow node 时：不是重新建图，而是在这个全局 graph 上按 node name、template name、ECU、别名、参数字段、refer-to 关系检索候选路径。

所以更准确的说法是：

```text
全量文档 -> 构建一次全局 KG/Graph
每个 flow node -> 在全局 KG 上查询相关 evidence path
```

面试可以这样说：

> Graph RAG 不是为每个 flow node 单独建图，而是先基于全量文档证据构建全局知识图谱。每个 flow node 来的时候，只是在图上查询相关实体和路径，得到这个节点自己的 evidence bundle。

## 9. Graph RAG 的原理是什么？

Graph RAG 的核心不是“向量相似度”，而是显式建模证据之间的结构关系。

它会把以下对象建成节点或关系：

- FlowNode / FlowTask。
- TemplateClass。
- ECU。
- DID / RID / ServiceID。
- Document / Section / Table / TableRow。
- ParameterField。
- FlowchartTitle / FlowchartImage。
- Evidence。
- AliasTerm。

对于每个 flow node，Graph RAG 会做几件事：

1. 生成 node alias：
   - node name。
   - template name。
   - 去掉 ECU 前缀后的 action name。
   - 一些业务别名，例如 DTC Read、DTC Clear、Change VMM。
2. 在 EvidenceUnit 里匹配候选证据：
   - 名称相似。
   - ECU 匹配。
   - template family 匹配。
   - 表格行有核心参数字段。
   - flowchart title 与模板类型匹配。
3. 抽取参数字段：
   - DID。
   - RID。
   - Request Parameter。
   - Source / Target。
   - Expected Value。
4. 生成 graph path：
   - `FlowTask -> uses_template -> TemplateClass`
   - `FlowTask -> supported_by -> TableRow/Section`
   - `TableRow -> row_has_field -> ParameterField`
   - `Section -> section_refers_flowchart -> FlowchartTitle`
5. 将 graph candidates 和 hybrid retrieval candidates 合并成节点级 evidence bundle。

面试可以这样说：

> dense/sparse 解决“文本能不能召回”，Graph RAG 解决“这个证据和当前 flow node 是什么关系”。它把 flow node、模板、ECU、参数字段、表格行和流程图路径连起来，所以最终 trace 里能看到为什么某条证据支持某个节点。

## 10. 输入给 LLM 生成代码时，上下文是如何编排的？

LLM 不是直接读取全部 PDF，也不是直接读取完整 workflow。每次只给一个 operation node 的受控上下文。

一个节点的 LLM 输入主要包括：

- operation 基本信息：
  - `step_key`
  - `order`
  - `row`
  - `column`
  - `node_name`
  - `template_name`
  - `class_name`
  - `script_type`
- allowed args：
  - 来自 template contract、模板示例、base workflow、evidence 字段。
- required args：
  - 来自 template contract。
- high-risk args：
  - `EcuBOMName`
  - `DID`
  - `RID`
  - `ServiceID`
  - `RequestParameter`
- base workflow reference：
  - 同名节点的 `ClassName`
  - 已有 args
  - 截断后的原始 XML
- template examples：
  - 同类 `ClassName` 或相似节点历史 XML 示例。
- evidence context：
  - `context_id`
  - `evidence_id`
  - `source_path`
  - `page_idx`
  - `bbox`
  - `parameter_fields`
  - `graph_paths`
  - `retrieval_source`
  - `retrieval_score`
  - 截断后的 evidence content

prompt 分三层：

- system prompt：定义模型角色，要求只能使用给定证据，不能编造 ECU/DID/RID/ServiceID。
- developer prompt：固定 XML 规范，要求输出一个 `ScriptNode`，ArgName 必须合法，高风险字段必须给 evidence。
- user prompt：放当前节点上下文、证据、模板契约、base workflow 和输出 JSON schema。

面试可以这样说：

> LLM 看到的是单节点 evidence package，而不是全量文档。上下文按 operation、contract、base workflow、template example、evidence context 分层编排，并且有 prompt budget 控制。这样既减少上下文长度，又能让每个参数有证据来源。

## 11. `average_generation_latency_seconds=17.770537s` 是什么延时？

这个指标是单个 LLM 节点生成的平均延时，不是整个 `flow.xlsx` 的总耗时。

当前验收里：

- `llm_generations = 11`
- `llm_valid_generations = 11`
- `average_generation_latency_seconds = 17.770537`
- `duration_seconds = 281.476869`

所以解释是：

- `17.77s`：平均每个 flow operation node 调一次 vLLM 的生成耗时。
- `281.48s`：完整 acceptance 总耗时，包含 service check、检索、LLM 生成、XML 渲染、workflow fusion、audit、regression、三组 scenario 对照等。

面试可以这样说：

> average generation latency 是 per-node LLM 调用平均耗时。完整验收耗时更长，因为它还包括向量检索、Graph RAG、XML 渲染、融合、审计、回归以及三组 scenario 对照。

## 12. deterministic guardrail 是什么意思？

deterministic guardrail 指的是确定性规则护栏。

大模型可以生成结构化 XML 计划，但不能让它最终决定高风险参数。系统会把模型原始输出和最终接受结果分开：

- `raw_llm_args`：模型原始生成的参数。
- `post_guardrail_args`：经过系统校验和修正后的参数。
- `guardrail_corrections`：记录每次修正的字段、原值、新值、证据、来源和原因。

重点保护字段：

- `EcuBOMName`
- `DID`
- `RID`
- `ServiceID`
- `RequestParameter`

当前结果里：

- raw LLM critical accuracy：`0.2`
- post-guardrail critical accuracy：`1.0`
- guardrail corrections：`12`

这说明模型原始输出不能直接信，但经过 evidence、模板和规则约束后，最终结果可以达到验收要求。

面试可以这样说：

> deterministic guardrail 就是让系统对高风险参数拥有最终决定权。LLM 可以建议，但 ECU、DID、ServiceID 这类字段必须经过 evidence、base workflow、deterministic plan 或 semantic default 支撑，不能由模型自由编造。

## 13. 每次 guardrail 是如何修正的？依据是什么？

guardrail 修正主要有三类依据。

### 13.1 deterministic plan

系统已有一套规则式 XML plan 生成逻辑，会从 Graph RAG evidence bundle 中抽取高风险参数。如果 LLM 生成的值缺失或与 deterministic plan 不一致，guardrail 会用 deterministic plan 的值覆盖。

例如：

- LLM 没填 DID。
- deterministic plan 从表格行抽到 `DID=0x3101`。
- 系统会把 DID 修正为 `0x3101`，并记录 correction source 为 `deterministic_plan`。

### 13.2 evidence context

如果 LLM 填了正确的值，但没有提供 evidence id，系统会尝试从当前 prompt context 中反查。

例如：

- LLM 填了 `EcuBOMName=ADCU1301`。
- 但没有给 evidence id。
- 系统发现某个 evidence context 的 source path、evidence id 或 parameter_fields 能匹配 ADCU1301。
- 系统补上 evidence id，correction source 为 `evidence_context`。

### 13.3 semantic default

有些业务字段在模板里有稳定默认值，但当前证据不一定每次显式出现。例如 DTC 相关流程中常见的 DID 或 ServiceID 默认值。系统可以用 semantic default，但必须 trace，不能伪装成 evidence 命中。

例如：

- 某类 DTC Read 默认 `DID=0x3101`。
- 某类 DTC Clear 默认 `ServiceID=0x19`。
- 系统可以使用，但 correction source 要写 `semantic_default`。

面试可以这样说：

> guardrail 修正不是拍脑袋改，而是有来源分类：deterministic plan、evidence context、semantic default。每次修正都会写 trace，这样可以区分“证据命中”和“语义默认值”，避免把默认值伪装成证据。

## 14. 渲染 operation XML 和 `serial_node.xml` 是什么意思？

LLM 生成的是节点级结构化计划，系统需要把这个计划渲染成标准 XML。

### 14.1 operation XML

operation XML 是每个 flow node 对应的单个 `ScriptNode` XML 文件。

例如一个节点可能被渲染为：

```xml
<ScriptNode Name="CMD1A1C_DTC_Read" ClassName="DTC_Read_Type1" ScriptType="NORMAL">
  <Args>
    <Arg ArgName="EcuBOMName">CMD1A1C</Arg>
    <Arg ArgName="DID">0x3101</Arg>
  </Args>
</ScriptNode>
```

项目会为 11 个 operation 分别输出 XML 文件，便于单节点查看、调试和回归。

### 14.2 `serial_node.xml`

`serial_node.xml` 是把这些 operation 按 `flow.xlsx` 的顺序组装成一个 `SerialNode`。

如果某一列/某一步里有多个并行节点，就会放进 `ParallelNode`。

当前验收里 serial XML 统计：

- root tag：`SerialNode`
- script nodes：`11`
- serial nodes：`1`
- parallel nodes：`3`
- args：`88`

面试可以这样说：

> operation XML 是单个节点的 ScriptNode，serial_node.xml 是把多个 ScriptNode 按 flow.xlsx 的串并行关系组装起来的局部流程片段。这样既能单节点调试，也能整体组装。

## 15. workflow fusion 是怎么融合的？

workflow fusion 的目标是把生成出来的 XML plan 融合进已有 base workflow，而不是重写整个 workflow。

融合过程大致是：

1. 读取 base workflow XML。
2. 遍历 base workflow 里的所有 `ScriptNode`，按 `Name` 建索引。
3. 遍历生成计划里的每个 planned node。
4. 如果 base workflow 中已经有同名节点：
   - 可以更新 `ClassName`，取决于配置。
   - 合并 Args。
   - 当前主要策略是 `fill_missing`：只填 base workflow 中为空或缺失的参数，不轻易覆盖已有值。
5. 如果 base workflow 中没有同名节点：
   - 根据配置决定是否插入。
   - 插入时会找到目标 `SerialNode`，按 flow 顺序追加。
   - 同一 order 的多个节点会放到 `ParallelNode` 里。
6. 输出 `fused_workflow.xml`。
7. 后续 audit 再比较 plan、base workflow、fused workflow 三者差异。

当前验收里的 fused workflow 结果：

- root tag：`PhaseNode`
- script nodes：`214`
- serial nodes：`30`
- parallel nodes：`7`
- args：`1457`

面试可以这样说：

> fusion 不是把 LLM 输出直接覆盖 base workflow，而是按 ScriptNode Name 找到对应节点，再按策略合并参数。默认更保守，只填缺失值，已有值冲突会进入 audit，而不是直接覆盖。

## 16. workflow audit 是做什么的？

audit 是最终验收前的重要校验。它比较三份东西：

- LLM/deterministic plan 想生成什么。
- base workflow 原来有什么。
- fused workflow 最终变成什么。

它会回答：

- plan 里的参数有没有真的写进 fused workflow。
- base workflow 里已有值和 plan 是否冲突。
- 多生成的 Arg 是可忽略，还是风险项。
- 是否需要人工 review。

当前优化后，系统把“可忽略的额外 Arg”和“真实冲突”分开：

- `extra_plan_arg_ignored`：base workflow 不需要、非必填、可安全忽略。
- `plan_conflict_kept_base`：plan 和 base 冲突，最终保留 base，需要关注。

当前 LLM scenario：

- audit review items：`2`
- resolution decisions：`88`
- auto resolved：`86`
- review required：`2`

面试可以这样说：

## 18. 实现中遇到的主要问题

### 18.1 默认 Python 环境缺依赖

问题：默认 `python` 没有 `pymilvus`、`sentence_transformers`。  
解决：固定使用 `.venv/bin/python` 跑 vector build/query 和 acceptance，并在 service check 中检查依赖。

### 18.2 Milvus/vLLM 服务未启动

问题：`19530` 和 `8008` 未监听时，真实验收 blocked。  
解决：acceptance runner 增加 Milvus TCP check、vLLM `/v1/models` check、embedding path check，不允许静默 fallback。

### 18.3 GPU0 被占用

问题：GPU0 被占用时，vLLM 只能先单卡启动。  
解决：先单卡 smoke test；GPU0 释放后切到双卡 TP2，并验证 `max_model_len=8192`。

### 18.4 embedding OOM

问题：Qwen3-Embedding 默认长序列，加上 vLLM 占 GPU，可能 OOM。  
解决：增加 `EMBEDDING_MAX_SEQ_LENGTH=512`、降低 batch size，必要时 query embedding 走 CPU。

这次 semantic hybrid LLM 联调里也遇到了同类问题：vLLM TP2 已经占用两张 GPU 后，如果 retrieval query embedding 继续放在 CUDA 上，会触发 OOM。实际验收时改为：

- vLLM 继续双卡 TP2，保证生成服务稳定。
- retrieval query embedding 使用 `EMBEDDING_DEVICE=cpu`。
- embedding 长度固定 `EMBEDDING_MAX_SEQ_LENGTH=512`。

面试答法：

> 生成模型和 embedding 模型同时抢 GPU 时，优先保证 vLLM 服务稳定。向量库已经提前离线构建，在线 query embedding 的吞吐压力较小，所以这轮验收把 query embedding 放到 CPU，避免和 TP2 vLLM 争显存。

### 18.5 LLM 原始关键参数准确率低

问题：raw critical accuracy 只有 `0.2`。  
解决：引入 deterministic guardrail，最终 post-guardrail critical accuracy 达到 `1.0`。

### 18.6 audit review items 初期偏高

问题：可忽略额外 Arg 和真实冲突混在一起。  
解决：新增 `extra_plan_arg_ignored` 分类，只把真实冲突留给 review。

### 18.7 acceptance 总耗时偏高

问题：总耗时约 `281s`。  
解决：已经记录 per-node token/latency；后续计划把 LLM 输出从 “JSON + 完整 XML” 收敛成 args plan，XML 由系统渲染。

## 19. 面试时可以主动强调的点

- 这个项目不是单纯 vLLM 部署，而是诊断资料解析、检索、生成、校验、融合、审计、回归的一整套闭环。
- LLM 不直接控制最终 workflow，只生成单节点计划。
- Milvus 只做语义召回，不作为事实源。
- Graph RAG 解决结构关系，dense/sparse 解决文本召回。
- raw LLM 输出和 post-guardrail 输出分开记录，能量化模型能力和系统修正能力。
- 所有高风险参数必须能追溯到 evidence、base workflow、deterministic plan 或 semantic default。
- acceptance 同时保留 sparse deterministic、hybrid deterministic、hybrid LLM XML 三组对照，能证明新能力没有破坏 baseline。
