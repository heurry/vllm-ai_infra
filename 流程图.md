2.3.1.3.5.7 DTC Clear Type7(Single DTC NRC Allowed)

流程说明
首先执行 DTC Read，也就是读取当前 ECU 中的故障码信息。
如果读取 DTC 后 ECU 返回 Positive Response，说明读取成功，流程进入下一步 DTC Clear。

在 DTC Clear 阶段，发送清除 DTC 的诊断请求。图右侧给出了相关诊断报文说明：

Req: 14 xx xx xx
Res: 54
refer to ecu TREG Parameter DTC List

含义是：

Req: 14 xx xx xx：发送 DTC 清除请求，服务 ID 为 0x14
Res: 54：如果清除成功，ECU 返回正响应 0x54
xx xx xx：表示需要清除的 DTC group 或参数，具体参考 ECU 的 TREG Parameter DTC List

发送 DTC Clear 后，判断 ECU 是否返回 Positive Response。

如果返回 Positive Response，说明 DTC 清除成功，流程进入 Pass。

如果没有返回正响应，则继续判断是否收到 Negative Response。
如果收到 Negative Response，说明 ECU 明确拒绝或无法执行清除操作，此时流程也进入 Pass。这里的逻辑表示：只要 ECU 有明确响应，不管是正响应还是负响应，都认为本次诊断通信流程完成。

如果既没有收到正响应，也没有收到负响应，说明可能出现了无响应、超时或通信异常。此时进入 Retry <= 3 判断。

当重试次数 小于等于 3 次 时，流程重新回到 DTC Clear，再次发送清除请求。

当重试次数 超过 3 次 后，认为 DTC Clear 操作失败，流程进入 Failed。

5. 总结描述
该流程用于执行带前置会话切换的 DTC 清除操作。流程首先发送 10 03 请求使 ECU 进入扩展诊断会话，若 ECU 返回 50 03 xx ... xx 正响应，则继续发送 14 FF FF FF 清除所有 DTC。若 DTC Clear 返回 54 正响应，则判定流程通过；若未收到正响应，则最多重试 3 次，超过重试次数后判定失败。若进入扩展会话阶段无响应或返回负响应，则直接判定失败。Thought for a few seconds这个流程图描述的是 IO Control 输入/输出控制测试流程，即在 ECU 进入扩展诊断会话并完成安全解锁后，发送 IO Control 指令控制某个执行器或信号，再读取控制结果，并判断是否符合预期。

1. 整体流程概括
该流程可以概括为：

进入扩展会话 → 安全访问解锁 → 启动 IO Control → 读取控制结果 → 判断结果是否可接受 → Pass / Failed

其中，进入扩展会话 和 安全访问 是前置条件；
Start IO Control 和 Read Result 阶段带有最多 3 次重试机制。

2. 进入扩展诊断会话
流程从 IO Control 开始，首先执行 Enter Ext-Session。
对应诊断报文为：
Req: 10 03Res: 50 03 xx ... xx
含义是：


10 03：请求进入扩展诊断会话；


50 03 xx ... xx：ECU 返回正响应，表示成功进入扩展会话；


后面的 xx ... xx 通常是会话保持时间、P2/P2* 等相关参数。


如果该步骤 无响应或返回负响应，流程直接进入 Failed。

3. Security Access Level 03 安全访问
进入扩展会话成功后，执行 Security Access Level 03。
对应报文为：
Req: 27 03Res: 67 03 seedReq: 27 04 keyRes: 67 04
含义是：


27 03：请求安全访问种子 seed；


67 03 seed：ECU 返回 seed；


测试端根据 seed 计算 key；


27 04 key：发送计算后的 key；


67 04：ECU 返回正响应，表示安全解锁成功。


如果安全访问阶段 无响应或返回负响应，流程直接进入 Failed。
因为未完成安全解锁时，ECU 通常不会允许执行 IO Control。

4. 启动 IO Control
安全访问成功后，执行 Start IO Control。
对应报文为：
Req: 2F XX XX 03 YYRes: 6F XX XX 03 YYDetail refer to ecu's TREG IO ID List
含义是：


2F：UDS 服务 InputOutputControlByIdentifier；


XX XX：需要控制的 IO Identifier，具体参考 ECU 的 TREG IO ID List；


03：IO Control 参数，通常表示某种控制类型，例如短期控制、强制控制等，具体以项目定义为准；


YY：控制值或控制参数；


6F XX XX 03 YY：ECU 返回正响应，表示 IO Control 请求被接受。


如果 Start IO Control 返回 Positive Response，则进入下一步 Read Result。
如果没有收到正响应，则进入 Retry <= 3 判断：


若重试次数 ≤ 3，则重新发送 Start IO Control；


若超过 3 次，流程进入 Failed。



5. 读取 IO Control 结果
Start IO Control 成功后，执行 Read Result，读取 ECU 当前状态或控制结果。
对应报文为：
Req: 22 XX XXRes: 62 XX XX ... XXDetail refer to ecu's IOID List
含义是：


22 XX XX：使用 UDS 0x22 服务读取指定 DID / IOID；


62 XX XX ... XX：ECU 返回读取结果；


返回数据中的状态值需要参考 ECU 的 IOID List 或诊断规范进行解析。


如果 Read Result 返回 Positive Response，则继续判断读取结果是否满足要求。
如果没有收到正响应，则进入第二个 Retry <= 3 判断：


若重试次数 ≤ 3，则重新执行 Read Result；


若超过 3 次，流程进入 Failed。

6. 结果判断
读取结果成功后，进入 Result Accept 判断。
如果读取到的 IO 状态、执行器反馈值或信号值符合预期，则进入 Pass。
如果读取结果不符合预期，则进入 Failed。
